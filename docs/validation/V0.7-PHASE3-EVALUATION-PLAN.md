# V0.7 Phase 3 — Simulation Evaluation Plan

Status: Planning (not started)

Scope: simulation evaluation infrastructure only

Architecture authority: `docs/design/ADR-003-SIMULATION-EVALUATION.md`

---

## 1. Purpose

Mem-D V0.7 Phases 1 and 2 delivered a complete simulation path. Simulation projects hypothetical store outcomes from recommendation outputs, is integrated into `analyze_file()`, and appears in JSON, Markdown, and terminal reports. Architecture audits for Phase 1 and Phase 2 are complete with **APPROVED WITH LIMITATIONS**.

| Phase | Responsibility | Status |
| --- | --- | --- |
| **Phase 1** | Simulation generation (`simulate_recommendations`, contracts, gold fixture, unit/property tests) | Complete |
| **Phase 2** | Simulation integration (pipeline, reporting, explainability enrichment) | Complete |
| **Phase 3** | Simulation evaluation (runner, metrics, artifacts, CI gate) | **Not started** |

Phase 3 exists to **validate simulation correctness**. Simulation is integrated but **not yet benchmark-certified**. Today:

- `tests/fixtures/simulation_gold.json` defines labeled expected outcomes
- `tests/test_simulation.py` runs binary pass/fail assertions and property tests
- Mem-D does **not** produce aggregated structural accuracy metrics, dimension-level accuracy rollups, or reproducible simulation benchmark artifacts comparable to recommendation, lifecycle, and evolution evaluation

**Phase 3 is evaluation-only.**

No simulation behavior changes are planned. Phase 3 does not alter merge/archive/review/keep semantics, application order, safety guards, metric formulas, pipeline ordering, or report rendering. It adds measurement, reporting, and regression gates on top of the existing simulation engine.

Simulation evaluation is required before V0.7 release certification and before V0.8 workflow architecture may proceed.

## 3. Evaluation Authority

**Sole ground truth:** `tests/fixtures/simulation_gold.json`

Each case `expected` block defines architecture-intended structural outcomes given known recommendation inputs. The evaluation runner compares produced `SimulationReport` against these labels.

**Explicitly not simulation ground truth:**

| Source | Role |
| --- | --- |
| **LongMemEval** | Regression posture guard only; no per-simulation labeled outcomes |
| **PERMA** | Regression posture guard only; no per-simulation labeled outcomes |
| **Recommendation benchmarks** (`recommendation_gold.json`, `recommendation_evaluation.json`) | Validates recommendation correctness (ADR-002); independent layer |
| **Lifecycle / evolution benchmarks** | Upstream capability validation; independent layers |
| **Analyze compression metrics** | Reference context only; not post-simulation equivalents |

**Rule:** Only `simulation_gold.json` determines simulation correctness. Regression guards may fail independently but **cannot override** gold fixture failures. A passing LongMemEval guard with failing gold cases is an evaluation failure.

## 4. Primary Evaluation Unit

### Authoritative object: `SimulationReport`

All quality judgments derive from comparing a produced `SimulationReport` against gold checkpoints for the same input `AnalysisReport`.

Evaluated fields include:

| Field group | Evaluation role |
| --- | --- |
| `simulatedMemories` / `simulatedMemoryCount` | Active store composition (`S_after`) |
| `simulatedMerges` | Merge projection structure (keeper, removedIds) |
| `simulatedArchives` | Archive log correctness |
| `simulatedReviewQueue` | Review and orphan-downgrade accounting |
| `simulationWarnings` | Safety warning correctness |
| `simulationId` | Determinism verification |
| `simulationMode` | Recorded in artifacts; must be `full` at launch |

### Supporting evidence: `SimulationMetrics`

`SimulationMetrics` supports validation but is **not** the primary evaluation unit.

Metrics are evaluated for:

- Consistency with store composition and event logs
- Consistency with hardened architecture formulas
- Internal structural relationships (e.g. trusted gain ≤ headline gain)

A case may pass store-composition checkpoints while failing metric-consistency checkpoints — that is a simulation failure.

## 5. Checkpoint Schema

Gold cases use an `expected` object with optional checkpoint fields. The evaluation runner evaluates **only fields present** in `expected` for each case, plus global safety invariants.

### Checkpoint classification

| Class | Meaning | Gates success |
| --- | --- | --- |
| **Required** | Must match when present in `expected` for a case tagged with that scenario type | Yes |
| **Optional** | Evaluated when present; absence from `expected` means no assertion | Yes, when present |
| **Diagnostic** | Recorded in artifacts; never gates success | No |

### Merge checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Active store size | `memoryCountAfter` | Required | Exact integer match |
| Active memory IDs | `activeMemoryIds` | Required | Set equality |
| Removed IDs | `removedIds` | Required | Set equality vs `S_before - S_after` |
| Merge groups simulated | `mergeGroupsSimulated` | Required | Exact integer |
| Keeper preserved | (derived) | Required | Keeper ID in `activeMemoryIds`; content unchanged vs source |
| Merge group structure | (derived) | Optional | `keeperId`, `removedIds` per group when gold adds `expectedMerges` (future fixture extension) |
| Keeper content unchanged | (derived) | Required | Property test + merge case verification |

### Archive checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Archives simulated | `archivesSimulated` | Required | Exact integer |
| Archive removal | `removedIds` | Required | Archive candidate absent from active store |
| Archive log entry | (derived) | Required | `simulatedArchives[].memoryId` matches; `archivedRecord.content` equals source |
| Lifecycle state captured | (derived) | Optional | `lifecycleState` on archive entry when gold specifies |

### Review checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Review queue size | `reviewQueueSize` | Optional | Exact integer |
| Unresolved review count | `unresolvedReviewCount` | Required | Exact integer |
| Conflict review count | `conflictReviewCount` | Optional | Exact integer |
| No structural removal | (derived) | Required | `memoryCountAfter == memoryCountBefore` for review-only cases |
| Review IDs preserved | (derived) | Required | All review-resolution IDs in active store |

### Keep checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| No structural change | `removedIds: []` | Required | Empty removal set |
| Unresolved count | `unresolvedReviewCount: 0` | Optional | Exact integer |
| Implicit keep fallback flag | `implicitKeepFallback` | Diagnostic | Records fallback path exercised |

### Warning checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Warning code | `warningCode` | Required | Code present in `simulationWarnings` |
| Warning count | `simulationWarningCount` | Required | Exact integer |
| Orphan downgrade | `orphanMergeDowngrade` | Required | Review queue entry with flag true |
| No unexpected warnings | (derived) | Required | Happy-path cases expect zero warnings |

### Explainability checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Fallback path exercised | `resolutionFallbackExplainability` | Diagnostic | Implicit keep without matching `Recommendation` |
| Explainability source | (derived) | Optional | `recommendation` vs `resolution_fallback` per event |
| Evidence refs non-empty | (derived) | Required | Every merge/archive/review event has ≥1 `evidenceRef` |
| Resolution ref prefix | (derived) | Required | At least one `resolution:` ref per simulated event |

### Metrics checkpoints

| Checkpoint | Field(s) | Class | Comparison |
| --- | --- | --- | --- |
| Memory count delta | `memoryCountDelta` | Optional | Exact integer |
| Duplicate reduction | `estimatedDuplicateReduction` | Optional | Exact integer |
| Utilization | (derived) | Diagnostic | `recommendationOutcomeUtilizationRate` recorded |
| Metric consistency | (derived) | Required | Formula checks (see §6, metric consistency accuracy) |
| Structural disclaimer | (derived) | Diagnostic | `metricsDisclaimer` present on report |

## 6. Evaluation Metrics

### Quality metrics (gate progression)

All quality metrics are computed from gold checkpoint pass/fail unless noted.

#### Overall structural accuracy

```
overall_structural_accuracy = passed_checkpoints / total_checkpoints
```

Where `total_checkpoints` = sum of evaluated checkpoint assertions across all gold cases (fields present in `expected` plus derived required checks for that case type).

**Primary quality gate.**

#### Merge projection accuracy

```
merge_projection_accuracy = passed_merge_checkpoints / total_merge_checkpoints
```

Evaluated on cases tagged merge: `sim_merge_1`, `sim_mixed_recommendation_set`, and merge portions of mixed cases.

#### Archive projection accuracy

```
archive_projection_accuracy = passed_archive_checkpoints / total_archive_checkpoints
```

Evaluated on `sim_archive_1`, archive portions of mixed cases.

#### Review preservation accuracy

```
review_preservation_accuracy = passed_review_checkpoints / total_review_checkpoints
```

Evaluated on `sim_review_1`, `sim_conflict_archive_merge`, and review-related assertions.

#### Warning accuracy

```
warning_accuracy = passed_warning_checkpoints / total_warning_checkpoints
```

Evaluated on `sim_orphan_merge_no_keeper`, `sim_duplicate_removal_skipped`, and happy-path zero-warning cases.

#### Orphan merge handling accuracy

```
orphan_merge_accuracy = passed_orphan_cases / total_orphan_cases
```

Case-level: all orphan checkpoints must pass for `sim_orphan_merge_no_keeper`.

**Hard gate:** `orphan_merge_accuracy == 1.0` (no partial credit).

#### Explainability accuracy

```
explainability_accuracy = passed_explainability_checkpoints / total_explainability_checkpoints
```

Events = all entries in `simulatedMerges`, `simulatedArchives`, `simulatedReviewQueue` across gold runs.

#### Metric consistency accuracy

```
metric_consistency_accuracy = passed_consistency_checks / total_consistency_checks
```

Per-case derived checks:

- `memoryCountDelta == memoryCountAfter - memoryCountBefore`
- `len(S_before) - len(S_after) == -memoryCountDelta` (sign convention)
- `estimatedDuplicateReduction` matches removable-slot formula on source clusters
- `estimatedTrustedCompressionGain <= estimatedCompressionGain`
- `recommendationsWithStructuralEffect == len(unique merge/archive recommendationIds in logs)`
- `unresolvedReviewCount` matches architecture formula

### Diagnostic metrics (do not gate success)

| Metric | Source | Purpose |
| --- | --- | --- |
| Recommendation utilization distribution | `recommendationUtilizationRate` per case | Posture drift |
| Outcome utilization distribution | `recommendationOutcomeUtilizationRate` | Structural effect rate awareness |
| Warning distribution | Count by `warning.code` | Safety path coverage |
| Projected reduction distribution | `memoryCountDelta`, `estimatedDuplicateReduction` | Structural posture summary |
| `simulationId` | Per case | Determinism smoke |
| Lifecycle distribution deltas | `lifecycleDistributionChange` | Diagnostic posture |

Diagnostic metrics appear in benchmark artifacts with `"diagnosticOnly": true`. They **must not** gate Phase 3 completion.

### Proposed CI gate thresholds (launch)

| Metric | Threshold |
| --- | --- |
| `overall_structural_accuracy` | `1.0` (all checkpoints pass on current 9-case fixture) |
| `orphan_merge_accuracy` | `1.0` |
| Safety property tests | 100% pass (mandatory, separate from accuracy) |
| `failures[]` | Empty |

Per-dimension accuracy (merge, archive, review, warning, explainability, metric consistency) reported alongside; launch threshold `1.0` on current fixture.

## 7. Float Tolerance Rules

ADR-003 requires explicit tolerance policy. Phase 3 adopts the following rules.

### Exact-match fields (integer or set)

No tolerance. Must match exactly.

| Field category | Examples |
| --- | --- |
| Counts | `memoryCountBefore`, `memoryCountAfter`, `memoryCountDelta`, `mergeGroupsSimulated`, `archivesSimulated`, `unresolvedReviewCount`, `simulationWarningCount` |
| Sets | `activeMemoryIds`, `removedIds` |
| IDs | `keeperId`, `removedIds[]`, `memoryId`, `warningCode` |
| Booleans | `orphanMergeDowngrade`, `conflictDetected` |
| Enum strings | `explainabilitySource` |

**Rationale:** Simulation is deterministic with discrete store operations. Integer and set outcomes should match exactly; floating-point ambiguity indicates a formula or rounding bug, not acceptable variance.

### Tolerance-based fields (float)

| Field | Tolerance | Rationale |
| --- | --- | --- |
| `estimatedStructuralCompressionBefore/After` | ±0.01 (absolute) | `_safe_percentage` rounds to 2 decimal places |
| `estimatedCompressionGain` | ±0.01 | Derived from rounded percentages |
| `estimatedTrustedStructuralCompressionBefore/After` | ±0.01 | Same rounding |
| `estimatedTrustedCompressionGain` | ±0.01 | Same rounding |
| `memoryReductionPercentage` | ±0.01 | Same rounding |
| `recommendationUtilizationRate` | ±0.0001 | Rounded to 4 decimal places in implementation |
| `recommendationOutcomeUtilizationRate` | ±0.0001 | Same |
| `referenceCompressionOpportunity` | Exact match | Copied from source report, not recomputed |

**Rationale:** Percentage fields use explicit `round(..., 2)` in simulation code. Tolerance matches rounding granularity, not statistical sampling error.

### Inequality guards (regression only)

| Guard | Rule |
| --- | --- |
| Trusted vs headline gain | `estimatedTrustedCompressionGain <= estimatedCompressionGain + 0.01` |
| Monotonic reduction | `memoryCountAfter <= memoryCountBefore` |
| Review preservation | review-resolution IDs ⊆ active IDs |

### Comparison implementation note

The evaluation runner should:

1. Compare integers and sets with `==`
2. Compare floats with `abs(a - b) <= tolerance`
3. Never compare `estimated*` metrics to LongMemEval/PERMA analyze compression figures

## 8. Safety Evaluation

Safety properties are **release-gating**. A single safety failure fails Phase 3 regardless of structural accuracy.

| Property | Validation method | Gate |
| --- | --- | --- |
| **Source immutability** | `AnalysisReport` snapshot before/after `simulate_recommendations`; object identity on memories, recommendations, resolutions | CI mandatory |
| **Idempotency** | Two runs produce identical `SimulationReport.model_dump()` and `simulationId` | CI mandatory |
| **Monotonic reduction** | `memoryCountAfter <= memoryCountBefore` on all gold cases | CI mandatory |
| **No orphan removal** | Every removed ID ∈ merge.removedIds ∪ archive.memoryId | CI mandatory |
| **Orphan safety** | Orphan case: zero removals + warning + downgrade | CI mandatory |
| **Review preservation** | Review resolutions never remove memories | CI mandatory |
| **Duplicate-removal safety** | `DUPLICATE_REMOVAL_SKIPPED` on labeled case | CI mandatory |

### Integration with evaluation runner

`evaluate_simulations()` must:

1. Run gold checkpoint comparison (accuracy layer)
2. Invoke or assert delegation to existing property tests in `test_simulation.py`
3. Report safety pass/fail separately from accuracy in benchmark artifacts

Safety failures appear in `failures[]` with `"category": "safety"` and block gate regardless of accuracy percentage.

## 9. Regression Guards

### LongMemEval and PERMA role

When simulation runs on published benchmark exports via `analyze_file()` (`simulationMode = full`):

| Guard | Condition | Type |
| --- | --- | --- |
| Trusted gain bound | `estimatedTrustedCompressionGain <= estimatedCompressionGain` | Inequality |
| Review-dominant posture | `unresolvedReviewCount / memoryCountBefore > resolutionsApplied / memoryCountBefore` | Inequality |
| Conditional merge effect | If `recommendationsWithStructuralEffect > 0` for merge, then `estimatedDuplicateReduction > 0` | Conditional |

### Rules

1. Guards are **regression indicators only** — posture and stability checks
2. Guards are **not accuracy sources** — no labeled expected outcomes on these exports
3. Guards **cannot override gold failures** — gold fixture failure always fails evaluation
4. Guard failures are reported as `"regressionGuardFailures"` in artifacts — warning severity unless maintainer elevates to gate
5. Guards do **not** assert equivalence to published LongMemEval or PERMA compression percentages

### Expected benchmark posture (documented, not gate at launch)

| Export | Expected simulation posture |
| --- | --- |
| LongMemEval cleaned | Review-heavy; low trusted structural gain relative to headline |
| PERMA user108 | Review-dominant; trusted gain << headline structural estimate |

Regression guard results recorded in `simulation_evaluation.md` under **Regression Guards (non-gating)** unless maintainer policy elevates them.

## 13. Relationship to Existing Evaluation Layers

Mem-D maintains **independent evaluation layers**. Each consumes its own gold fixture and answers a different question.

| Layer | ADR | Fixture | Question |
| --- | --- | --- | --- |
| Lifecycle inference | — | `lifecycle_gold.json` | Did lifecycle assignment match gold? |
| Evolution auditing | — | `evolution_gold.json` | Did evolution signals match gold? |
| Recommendation resolution | ADR-002 | `recommendation_gold.json` | Did `resolvedAction` match gold? |
| **Simulation projection** | **ADR-003** | **`simulation_gold.json`** | **Did `SimulationReport` match gold?** |

### Separation rules

1. **ADR-002 validates recommendation correctness.** High recommendation accuracy does not imply simulation pass.
2. **ADR-003 validates simulation correctness.** Simulation pass does not imply recommendations were correct.
3. **Neither replaces the other.** Both must pass before workflow architecture.
4. **Failure isolation.** A lifecycle regression does not automatically fail simulation evaluation unless shared inputs change gold labels (separate investigation).
5. **Metric namespace separation.** Recommendation metrics use `resolvedAction` labels; simulation metrics use store composition and event logs. No shared accuracy denominator.

## Evaluation Runner Plan

**Module:** `memd/benchmarks/simulation_evaluation.py` (mirrors `memd/benchmarks/recommendation_evaluation.py`)

### `evaluate_simulations(...)`

**Inputs:**

| Parameter | Type | Default |
| --- | --- | --- |
| `gold_path` | `Path` | `tests/fixtures/simulation_gold.json` |
| `build_report` | callable | Shared helper from `test_simulation.py` or extracted fixture module |

**Processing steps:**

1. Load and validate gold fixture shape
2. For each case: build `AnalysisReport` via shared helper
3. Call `simulate_recommendations(report)`
4. Compare `SimulationReport` against case `expected` checkpoints
5. Run derived checks (keeper content, explainability refs, metric consistency)
6. Aggregate quality metrics
7. Collect failures with `{caseId, checkpoint, expected, actual, category}`
8. Return `SimulationEvaluationResult` dataclass

**Outputs:** Structured result + optional JSON/Markdown via script.

### Shared helpers

Extract `build_report_from_case()` and `load_simulation_cases()` from `tests/test_simulation.py` into a shared module (e.g. `memd/benchmarks/simulation_fixtures.py`) to prevent runner/test drift. **No duplication of simulation logic.**

---

## Explicit Non-Goals

Phase 3 does **not** include:

| Excluded item | Rationale |
| --- | --- |
| Simulation behavior changes | Evaluation-only phase |
| Workflow architecture or execution | V0.8; gated on evaluation |
| Action systems / automation | V1.0 |
| Memory modification or persistence | Out of scope per AGENTS.md |
| Recommendation redesign | ADR-001/002 owned separately |
| New simulation modes in engine | Optional future; not required for evaluation launch |
| Re-clustering simulated stores | Architecture non-default |
| PERMA/LongMemEval ground-truth labeling | Regression guards only |
| Conflating simulation accuracy with recommendation accuracy | ADR-003 separation |

## Risks

| Risk | Mitigation |
| --- | --- |
| **Gold fixture bias** | 9 synthetic cases; document as architecture-compliance benchmark |
| **Runner/test drift** | Shared fixture loading module; single source of truth |
| **Metric misinterpretation** | Diagnostic metrics labeled non-gating; disclaimers in artifacts |
| **False confidence from small sample** | Report checkpoint counts alongside percentages |
| **Float tolerance ambiguity** | Document exact rules in §7 and `SIMULATION-BENCHMARK.md` |
| **Regression guard elevation** | Default non-gating; explicit maintainer policy to elevate |
| **Phase scope creep** | Evaluation-only; no simulationMode CLI or engine changes |
