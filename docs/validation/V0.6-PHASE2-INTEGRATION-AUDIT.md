## Verification Details

### 1. Recommendation generation invoked exactly once per analysis run — **Pass**

**Evidence:**

- `plan_recommendations` is called in exactly one production path: `memd/pipeline.py` inside `analyze_file()`, after `apply_policy()` and before `AnalysisReport` construction.
- Repository grep shows no other `plan_recommendations` call sites in `memd/` (CLI uses `analyze_file` only).
- Call order matches architecture: `plan_governance_actions` → `apply_policy` → `plan_recommendations` → `AnalysisReport`.

```63:70:memd/pipeline.py
    recommendations, memory_resolutions, recommendation_summary = plan_recommendations(
        records,
        clusters,
        validation,
        insights,
        actions,
        metrics=metrics,
    )
```

**Conclusion:** One recommendation pass per analysis run. No duplicate or nested invocation.

---

### 2. Recommendation outputs correctly propagated into `AnalysisReport` — **Pass**

**Evidence:**

- `AnalysisReport` extended with `recommendations`, `memoryResolutions`, and `recommendationSummary`, all with safe defaults (`()`, empty summary).
- Pipeline assigns all three return values from `plan_recommendations` into the report constructor.
- Integration test `test_pipeline_recommendation_behavior_matches_phase1_isolation` asserts pipeline output equals an isolated `plan_recommendations(...)` call with the same inputs.

**Conclusion:** Propagation is complete and verified by tests.

---

### 3. JSON serialization complete and stable — **Mostly pass**

**Evidence:**

- `report_to_dict` adds `recommendations`, `memoryResolutions`, and `recommendationSummary` after existing keys (`metrics`, `actionSummary`, `policySummary`, `actions`, `clusters`, `categories`, `validation`, `insights`).
- `recommendation_to_dict` serializes all contract fields: `recommendationId`, `action`, `subtype`, `confidence`, `reason`, `affected_memories`, `evidence`, `estimatedImpact`, `priority`, `requiresHumanApproval`, `sourceActionIds`, `sourceInsightIds`, `suppressedCandidates`, `conflictDetected`.
- `memory_resolution_to_dict` and `recommendation_summary_to_dict` cover all `MemoryResolution` and `RecommendationSummary` fields.
- Recommendation tuples are built with sorted memory IDs and `recommendation_sort_key` (priority, then `recommendationId`), supporting deterministic ordering.

**Gaps:**

| Finding | Severity | Detail |
| --- | --- | --- |
| `affected_memories` uses snake_case in JSON | Low | Architecture and other JSON keys use camelCase (`memoryId`, `sourceActionIds`). Contract field name leaks into serialization. |
| No golden JSON snapshot test | Low | Integration tests assert key presence and evidence shape, not full serialized payload stability across runs. |

**Conclusion:** Serialization is functionally complete. Stability is supported by deterministic engine ordering but not locked by snapshot tests.

- Grouped subsections: Merge, Archive, Review; Keep only when keep records with evidence exist.

- Summary rollup includes total, per-action counts, and `memoryResolutionCount`.

**Gap:**

| Finding | Severity | Detail |
| --- | --- | --- |
| Section title differs from architecture | Low | Architecture specifies `## Governance Recommendations`; implementation uses `## Recommendation Summary`. Content structure and placement align; title only. |

**Conclusion:** Rendering behavior matches architecture intent; section naming is a minor cosmetic deviation.

### 5. Terminal rendering does not expose incomplete recommendation records — **Pass**

**Evidence:**

- `displayable_recommendations()` requires non-empty `evidence` and excludes `KEEP`.
- Terminal table uses `displayable[:5]` only when the filtered list is non-empty.
- Table columns: action, confidence, reason (truncated), evidence summary (`format_evidence_summary`).
- Rollup line shows merge / archive / review counts from `recommendationSummary`.
- Implicit `keep` resolutions are not shown as recommendation rows (correct per architecture noise-reduction default).

**Conclusion:** Terminal surfaces only evidence-backed, non-keep group recommendations.

### 6. Recommendations without evidence are never emitted — **Pass**

**Evidence:**

- Engine filter before return:

```138:142:memd/recommendations.py
    recommendations = tuple(
        recommendation
        for recommendation in recommendations
        if recommendation_has_evidence(recommendation)
    )
```

- `recommendation_has_evidence` returns `bool(recommendation.evidence)`.
- Reporting layer applies a second filter via `displayable_recommendations` for Markdown and terminal (defense in depth).
- `test_recommendations_include_evidence` asserts every `report.recommendations` entry has evidence, reason, and action.

**Note:** `MemoryResolution` records may exist without a corresponding group `Recommendation` (e.g. implicit `keep`). That is architecture-compliant; the criterion applies to `Recommendation` records, not resolutions.

**Conclusion:** No evidence-empty recommendations are emitted from `plan_recommendations`.

## Recommended Fixes

Audit-only recommendations (no code changes in this audit):

### Phase 2 follow-up (optional, low priority)

1. **Align Markdown section title** with architecture (`## Governance Recommendations`) or update architecture doc to match implemented title.
2. **Serialize `affectedMemories` in camelCase** in `recommendation_to_dict` for JSON consistency (or document snake_case as intentional contract-field mirror).
3. **Add upstream isolation regression test** — run pipeline steps through `apply_policy`, capture `actions`/`policySummary`/`validation` hashes, assert identical with and without recommendation assignment (proves no side effects).

### Phase 3 (planned)

4. **Wire or remove `insights` parameter** — attach insight IDs and evidence, or defer parameter until needed.
5. **Expose `--include-keep` CLI flag** per architecture.
6. **Add JSON snapshot test** for a fixed fixture to lock serialization stability.

### Phase 4 (planned)

7. **Recommendation gold evaluation runner** and benchmark baseline sections for recommendation counts.
8. **Enrich conflict review evidence** further per Phase 1 audit (full merge + archive + cluster trust bundle on all conflict escalations).
