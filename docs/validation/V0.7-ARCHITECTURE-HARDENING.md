## Rationale

### M1 — Structural estimates vs benchmark metrics

LongMemEval and PERMA publish `compressionOpportunity` from `calculate_metrics()` on analyzed stores. Post-simulation values cannot claim equivalence without re-clustering. Hardening defines parallel **structural** removable-slot math using source cluster maps adjusted for `S_after`, explicitly labeled `estimated*` and disclaimed in `SimulationReport.metricsDisclaimer`.

### M2 — Explainability without full Recommendation coverage

V0.6 emits `MemoryResolution` for every memory but filters evidence-bearing `Recommendation` records. Implicit keeps are valid production outputs. Fallback explainability from resolution + validation prevents implementation failures and false-negative completeness tests.

### M3 — Orphan-merge as safety, not re-resolution

Strict consumption of merge resolutions without a keeper would remove memories unsafely. The guard refuses removal, emits a warning, and accounts the case as unresolved review in simulation metrics only — preserving ADR-001 outputs for audit integrity.

### M4 — Utilization from recommendation outcomes

Governance actions are cluster-scoped and include types (e.g. `review_cluster`) that do not map 1:1 to simulated structural changes. Archive flows from lifecycle resolutions, not governance action types. Recommendation-outcome utilization aligns with simulation's actual inputs and outputs.

## Remaining Implementation Risks (post-hardening)

| Risk | Severity | Mitigation in architecture |
| --- | --- | --- |
| Structural compression estimates may still diverge from intuitive "re-analyze" expectations | Medium | `metricsDisclaimer` + report labeling; optional future re-analysis mode documented as non-default |
| `removable_estimate` using source cluster maps on `S_after` may over/under-estimate if cluster topology would change after merge | Low | Accepted V0.7 tradeoff; documented as structural not topological |
| Fallback reason templates may feel generic for complex review cases | Low | evidenceRefs carry detailed signal links; templates are minimum bar |
| Large stores: `simulatedMemories` tuple size in report | Low | Implementation may add summary-only mode later; out of V0.7 scope |
| Hash stability across Python versions for `simulationId` | Low | Use canonical JSON serialization in implementation |

## Final Recommendation

**V0.7 Phase 1 implementation may begin.**

Architecture hardening resolves all medium-severity audit findings. Layer boundaries, safety guarantees, and ADR alignment are unchanged. Implementation should follow the hardened architecture document verbatim for metrics, explainability fallback, orphan-merge handling, and utilization formulas.

Suggested Phase 1 scope:

1. `simulate_recommendations(report) -> SimulationReport` core module
2. Contract models per hardened schemas
3. Unit tests + `simulation_gold.json` structural fixtures
4. Property tests for safety invariants

Report serialization and CLI integration may follow in Phase 1.1 per existing architecture notes.
