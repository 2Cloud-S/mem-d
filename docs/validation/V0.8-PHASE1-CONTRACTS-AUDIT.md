## Findings

### High

None.

### Medium

- **M1 — Missing item-level guard for non-review actions carrying review queues**
  - **Evidence:** `memd/contracts.py` `WorkflowItem.validate_structure` does not enforce action/reviewRequirement consistency; review queue assignment can be structurally valid even for non-review actions if `reviewRequirement.required=True`.
  - **Impact:** Weakens a stated Phase 1 invariant ("non-review workflow items must not be assigned review queues"), leaving room for semantically invalid plans.
  - **Resolution:** Add structural validator ensuring non-review actions cannot carry review queue/subtype requirements.
  - **Blocks Phase 1 approval:** No (limitations).
  - **Blocks Phase 2 planning:** No.
  - **Blocks Phase 2 implementation:** **Yes, should be fixed before implementation**.

- **M2 — Incomplete negative test coverage for invalid enum values across closed domains**
  - **Evidence:** `tests/test_workflow_contracts.py` validates many invariants but lacks explicit invalid-value tests for several enum-bearing fields (for example planner/operator status and some action/status closed-domain paths).
  - **Impact:** Reduced confidence that deserialization hard-fails uniformly on all enum contracts.
  - **Resolution:** Add dedicated invalid-enum-value tests per key enum field families.
  - **Blocks Phase 1 approval:** No (limitations).
  - **Blocks Phase 2 planning:** No.
  - **Blocks Phase 2 implementation:** No, but strongly recommended pre-Phase 2 implementation hardening.

### Low

- **L1 — Whitespace-only target-ID hardening is unevenly covered by tests**
  - **Evidence:** IDs like `stepId`/`targetId` are normalized and validated, but tests do not explicitly assert whitespace-only rejection for operation target arrays (`archiveTargetIds`, `reviewTargetIds`, `removableIds`).
  - **Impact:** Minor assurance gap.
  - **Resolution:** Add targeted negative tests and optional stricter validators for per-target string normalization.
  - **Blocks Phase 1 approval:** No.
  - **Blocks Phase 2 planning/implementation:** No.

- **L2 — Scope traceability could be clearer in implementation docs**
  - **Evidence:** Current implementation report states outcomes correctly but does not explicitly tabulate all non-modified protected files.
  - **Impact:** Minor audit traceability issue only.
  - **Resolution:** Add a compact "scope verification table" in implementation docs (optional).
  - **Blocks:** No.
