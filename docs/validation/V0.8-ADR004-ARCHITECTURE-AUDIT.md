Date: 2026-06-19  
Auditor role: Independent, read-only architecture audit  
Method: Cross-check `ADR-004-WORKFLOW-ARCHITECTURE.md` against ADR-001/002/003, V0.7 simulation architecture, release-readiness and external-validation evidence, and live `memd/` contracts and pipeline modules

## 1. Executive Summary

ADR-004 defines a **read-only, deterministic workflow planning layer** that packages upstream recommendation, policy, governance, and simulation outputs into reviewable plans with explicit planner/operator separation, structured step operations, SHA-256 identity, and a fail-closed aggregate status model.

Independent review finds the ADR **architecturally sound and compatible** with the V0.7 codebase. Pipeline placement after simulation is correct. Proposed contracts map cleanly to existing `Recommendation`, `MemoryResolution`, `GovernanceAction`, `PolicyDecision`, `SimulationReport`, and role conventions (`keeper`, `removable`). Execution, persistence, and scheduling are explicitly excluded. V0.8 / V0.9 / V1.0 boundaries are clear.

Primary gaps are **specification–implementation alignment** items that should be resolved in Phase 1 contracts (not architecture blockers):

- Simulation identity check **R2** uses a broader fingerprint than `memd/simulation.py::_compute_simulation_id()` today.
- `SimulationReport.analysisReportRef` exists in contracts but is **never populated** by the simulation engine.
- Plan-level `INPUT_INTEGRITY` without item-level blocked status may not surface a blocked-family aggregate label.
- Review subtype detection and `simulationRefs` key format need Phase 1 normative definitions.

The revised aggregate precedence (§6.3.4) correctly prevents vacuous `ready_for_execution`, requires `structural_count >= 1`, and separates `mixed_blocked` from `mixed_blocked_review` and `all_blocked`.

**Verdict: APPROVED WITH LIMITATIONS**

## 5. State Model Validation

### 5.1 Planner / operator separation

| Check | Result |
| --- | --- |
| `plan_workflows()` sets `operatorItemStatus == none` only | **Specified** (§6.2) |
| `approved` / `rejected` / operator `deferred` only via `apply_workflow_decisions()` | **Specified** |
| Initial plan cannot satisfy H2 (`operatorItemStatus == approved` on structural) | **Correct — `ready_for_execution` impossible on initial plan** |
| `keep` never eligible for handoff | **Specified** (§6.2, H1–H2) |

**Note:** §1 states `plan_workflows()` never emits `approved`, `rejected`, or `ready_for_execution`. Aggregate `ready_for_execution` is only reachable after `apply_workflow_decisions()` because H2 requires structural operator approval. Recommend adding an explicit invariant in Phase 1: **initial plan `aggregateStatus` never equals `ready_for_execution`**.

### 5.2 Truth-table analysis

Predicates evaluated against §6.3.4 precedence (first match wins). `handoff_ready` requires H1–H7.

| Scenario | Key predicates | Winning rule | `aggregateStatus` | `ready_for_execution`? |
| --- | --- | --- | --- | --- |
| **Empty** | `totalItems=0`, `keepCount=0` | 1 | `empty` | No (H1) |
| **All keep** (summary-only) | `totalItems=0`, `keepCount>0` | 2 | `all_keep` | No (H1) |
| **Review only** (initial) | `any_unresolved_review`, no blocked | 15 or 16 | `requires_review` | No (H1) |
| **Deferred only** (initial planner) | `plannerItemStatus=deferred`, op `none` | 17 | `proposed` | No |
| **Deferred only** (all op deferred) | `all_actionable_deferred` | 8 | `deferred` | No (H1) |
| **Rejected only** | `all_actionable_rejected` | 7 | `rejected` | No (H1) |
| **All blocked** | `all_actionable_blocked` | 4 | `all_blocked` | No (H5) |
| **Blocked + proposed** | `has_mixed_blocked_items`, no unresolved review | 6 | `mixed_blocked` | No (H5) |
| **Blocked + approved structural** | same | 6 | `mixed_blocked` | No (H5) |
| **Blocked + operator-deferred** | same | 6 | `mixed_blocked` | No (H5) |
| **Blocked + unresolved review** | `any_item_blocked` ∧ `any_unresolved_review` | 5 | `mixed_blocked_review` | No (H5) |
| **Approved structural only** (all structural approved, no review/defer items, full sim) | `handoff_ready` | 9 | `ready_for_execution` | **Yes** |
| **Approved structural + unresolved review** | `any_structural_approved` ∧ `any_unresolved_review` | 10 | `partially_approved` | No (H4) |
| **Partially approved structural** (some structural `none`) | rule 11 or 12 | `partially_approved` | No (H2) |
| **Missing simulation** | `planningMode=recommendations_only` | 9 fails H7; typically 13–17 | `approved` / `proposed` / `requires_review` | **No** (H7) |
| **Plan-level integrity blocker** (mixed items, no item blocked) | `has_plan_blocker`, not rules 4–6 | 15 or 17 | `requires_review` or `proposed` | No (H6) |
| **Plan-level integrity + item blocked + review** | rules 4–6 | `mixed_blocked_review` or `mixed_blocked` | No | |
| **Review-only all operator-resolved** | `structural_count=0`, all resolved | 13 | `approved` | No (H1) |
| **All-keep emitted items** | all `action==keep` | 3 | `all_keep` | No (H1) |

### 5.3 Confirmations

| Requirement | Verified |
| --- | --- |
| Exactly one status per case (deterministic first-match) | **Yes**, given defined predicates |
| `ready_for_execution` requires explicit structural approval | **Yes** — H2 requires `operatorItemStatus == approved` on every structural item |
| `structural_count >= 1` for readiness | **Yes** — H1 |
| Vacuous readiness impossible (review-only, all-deferred, all-rejected, all-keep, empty) | **Yes** — H1 fails when `structural_count == 0` |
| Keep never executable | **Yes** — excluded from `structural` and eligibility |
| Blocked items cannot be approved | **Yes** — §5.7 invariants + H5 |
| Unresolved reviews prevent readiness | **Yes** — H4 |
| Blocked + non-blocked ≠ `all_blocked` | **Yes** — rule 6 `mixed_blocked` |

### 5.4 Residual state-model gap

**Plan-level `INPUT_INTEGRITY` with no item-level `blocked`:** precedence may yield `requires_review` or `proposed` rather than a blocked-family status, even though `handoff_ready` is blocked via H6. Functionally fail-closed for execution; **aggregate label may understate severity** (see Finding M3).

## 10. Findings by Severity

### High

*None.* No architecture blockers or unsafe execution ambiguity identified. Workflow planning is explicitly read-only; `ready_for_execution` is planning handoff intent only; V1.0 execution requires a separate ADR.

---

### Medium

#### M1 — Simulation fingerprint mismatch (R2 vs implementation)

- **Evidence:** ADR §4 R2 and §12 include simulation events and warnings in the simulation fingerprint. `memd/simulation.py::_compute_simulation_id` hashes only memories, resolution subset, mode, and policy profile (`simulation.py:706–723`).
- **ADR section:** §4 (R2), §12
- **Impact:** Strict R2 implementation would emit `INPUT_INTEGRITY` on every plan against current V0.7 reports unless algorithms are aligned.
- **Recommended resolution:** Phase 1 must either (a) normatively adopt the existing `_compute_simulation_id` payload as R2, or (b) extend simulation ID computation and bump simulation evaluation fixtures. Document choice in contracts.
- **Blocks ADR acceptance?** **No** — requires documented alignment decision before Phase 2.

#### M2 — `analysisReportRef` never populated

- **Evidence:** `SimulationReport.analysisReportRef` defaults to `""`; `simulate_recommendations()` does not set it (`simulation.py:206–218`, `contracts.py:364`).
- **ADR section:** §4 R3
- **Impact:** R3 is dormant; stale cross-layer detection relies entirely on workflow-computed `sourceAnalysisRef` vs replanning.
- **Recommended resolution:** Phase 1 or V0.8 Phase 3 should specify whether simulation populates `analysisReportRef` or R3 remains optional with workflow-only fingerprints.
- **Blocks ADR acceptance?** **No**

#### M3 — Plan-level integrity aggregate labeling

- **Evidence:** §6.3.4 — `has_plan_blocker` blocks `handoff_ready` (H6) but has no dedicated aggregate rule; plans with integrity failure and no item-level `blocked` may show `proposed` or `requires_review`.
- **ADR section:** §6.3.3–§6.3.4
- **Impact:** Reporting may understate integrity failures; execution still blocked.
- **Recommended resolution:** Add optional aggregate rule (e.g. `integrity_blocked`) or precedence rule before rule 15 when `has_plan_blocker` and not caught by 4–6.
- **Blocks ADR acceptance?** **No**

#### M4 — `simulationRefs` key format undefined

- **Evidence:** §5.2 requires `simulationRefs`; R4 requires resolution to `simulationReport` records; no canonical key encoding (contrast simulation explainability `warning:ORPHAN_MERGE_NO_KEEPER` in `simulation.py:520`).
- **ADR section:** §5.2, §4 R4, §9
- **Impact:** Non-deterministic or incompatible refs across planner/reporting/evaluation.
- **Recommended resolution:** Define normative ref strings in Phase 1 (e.g. `merge:{recommendationId}`, `archive:{memoryId}`, `warning:{code}:{memoryId}`).
- **Blocks ADR acceptance?** **No**

#### M5 — Review subtype detection underspecified

- **Evidence:** Subtype catalog §5.6 lists 11 subtypes; routing table §7.5 maps subtypes to queues; **detection rules** from `validation` / `Recommendation.subtype` / evidence signals are not fully normative.
- **ADR section:** §5.6, §7.5, §8
- **Impact:** Planner implementations may diverge; gold fixture disputes.
- **Recommended resolution:** Phase 1 appendix mapping subtype → upstream signal predicates (reuse ADR-001 modifier names where possible).
- **Blocks ADR acceptance?** **No**

#### M6 — `ACTION_PRIORITY_RANK` not declared

- **Evidence:** §5.5 references `ACTION_PRIORITY_RANK[recommendationPriority]`; constant not in contracts or ADR normative appendix. Ordering exists in `recommendations.py` for tie-breaking.
- **ADR section:** §5.5, §7.4
- **Impact:** `queueRank` base could drift from recommendation engine ordering.
- **Recommended resolution:** Add explicit rank table to Phase 1 contracts doc.
- **Blocks ADR acceptance?** **No**
