# V0.8 Phase 4 Workflow Evaluation Audit

Date: 2026-06-23

## Executive Summary

Phase 4 workflow evaluation is implemented and runnable. The focused workflow
evaluation tests pass, the runner regenerates artifacts, touched-file Ruff
passes, and the workflow regression suite passed during runtime validation. The
current benchmark artifact reports:

```text
caseCount: 33
checkpointCount: 113
all quality metrics: 1.0
all 18 safety gates: pass
gatePassed: true
```

However, this audit found architecture-compliance defects that the green tests
and artifacts do not catch. In particular:

1. safety gates for source immutability and `no_effects` are too weak to satisfy
   ADR-005;
2. review-subtype non-vacuity can be satisfied by non-scored metadata
   (`coverageSubtypes`) rather than actual exact workflow output checkpoints;
3. duplicate checkpoint handling in the metric aggregator records a failure but
   does not fail fixture validation before scoring as ADR-005 requires.

These defects affect benchmark certification trust. Phase 4 is therefore not yet
approved for V0.8 release certification.

Final verdict:

```text
NOT APPROVED
```

## ADR-005 Compliance

ADR-005 requires independent workflow evaluation with:

- `tests/fixtures/workflow_gold.json` as sole workflow correctness authority;
- exact checkpoint accounting;
- non-zero denominators for all quality metrics;
- non-vacuous safety gates;
- canonical checkpoint identity;
- deterministic artifacts;
- no execution, persistence, scheduling, approval UX, or external scoring.

The implementation partially satisfies these requirements:

- The evaluator exposes `evaluate_workflows(gold_path: Path | None = None)`.
- The runner calls the same evaluator API used by tests.
- The fixture is workflow-specific and separate from recommendation/simulation
  gold fixtures.
- The artifact includes the required high-level result contract fields.
- Metrics are separated from diagnostics.
- The implementation does not modify planner, contracts, pipeline, reports,
  recommendations, simulation, policy, or actions.

Material ADR-005 gaps remain:

- safety instrumentation does not fully implement ADR-005 boundaries;
- source immutability is not checked against a pre-call snapshot;
- review-subtype non-vacuity can pass through metadata not compared to actual
  workflow output;
- duplicate checkpoint rejection is not enforced before scoring in all paths.

## Ground Truth and Fixture Validation

`tests/fixtures/workflow_gold.json` is the named workflow ground truth authority.
The evaluator does not read recommendation gold, simulation gold, external
validation artifacts, or committed benchmark artifacts as labels.

Fixture builders in `_build_report()` construct input `AnalysisReport` objects.
The builder implementation does not call:

- `plan_workflows()`;
- `apply_workflow_decisions()`;
- `compute_aggregate_status()`;
- production routing helpers.

No builder-derived hidden expected workflow output was found.

One hidden-authority risk remains outside the builders: exact review subtype
coverage can be satisfied through `coverageSubtypes` metadata in the fixture.
Those values are counted by fixture validation but are not compared against
actual planner output as `review_routing` checkpoints. This weakens the fixture's
role as concrete workflow correctness authority for those subtype paths.

## Findings

### High

#### H1 - Safety gates are not certification-grade

**Evidence:** In `memd/benchmarks/workflow_evaluation.py`,
`source_report_immutability` compares `runtime.report.model_dump()` to
`copy.deepcopy(runtime.report).model_dump()` after the initial planning run.
This does not compare against a pre-planning snapshot and cannot detect mutation
that already occurred during planning. The `no_effects` guard patches
`socket.socket` and `subprocess.Popen`, but does not instrument filesystem
writes, action execution APIs, persistence/scheduling APIs, or source object
mutation. ADR-005 requires bounded CI instrumentation for those boundaries.

**Impact:** The artifact reports every safety gate passing, but two hard-gating
safety claims are under-instrumented. This invalidates benchmark certification
until fixed.

**Required disposition:** Capture source report snapshots before every
planner/decision call and compare after the call. Extend bounded `no_effects`
instrumentation to detect filesystem writes inside the planner/decision boundary
and calls to action execution, persistence, and scheduling APIs, or explicitly
wire sentinels for the project-level APIs that represent those effects. Add
tests that prove these sentinels fail when violated.

**Blocks Phase 4 certification:** Yes  
**Blocks V0.8 release readiness:** Yes

### Medium

#### M1 - Review subtype non-vacuity can be satisfied by non-scored metadata

**Evidence:** Fixture validation counts `coverageSubtypes` in
`expected.reviewRoutes`, but `_evaluate_checkpoint()` compares only the actual
`subtypes` field. The fixture uses this mechanism for values such as
`policy_blocked`, `lifecycle`, and `simulation_safety` where the exact route
expectation does not include the subtype in actual planner output.

**Impact:** ADR-005 requires every review subtype to appear at least once in a
`review_routing` checkpoint. The current implementation can pass non-vacuity
without scoring those subtypes as actual workflow output. This weakens the
`reviewRoutingAccuracy` denominator and fixture coverage claim.

**Required disposition:** Either construct production-valid cases that actually
emit each review subtype in scored `review_routing` checkpoints, or explicitly
amend the architecture/fixture schema to treat unreachable subtype values as
version-specific required-absence/disposition checks similar to reserved blocker
codes. Do not count non-scored metadata as exact subtype coverage.

**Blocks Phase 4 certification:** Yes  
**Blocks V0.8 release readiness:** Yes

#### M2 - Duplicate checkpoint rejection is not consistently pre-scoring

**Evidence:** `_validate_fixture()` rejects duplicate expanded checkpoints, but
`_quality_metrics()` also contains duplicate handling that records a scored
failure and continues. The dedicated test directly exercises `_quality_metrics()`
and accepts a failure record rather than proving fixture-invalid behavior from an
expanded duplicate fixture.

**Impact:** ADR-005 requires duplicate checkpoint rejection before scoring. The
implementation has the correct fixture-validator path, but tests and fallback
aggregation semantics could allow future code paths to treat duplicates as
scored failures instead of fixture-invalid errors.

**Required disposition:** Add an explicit invalid-fixture test that creates
duplicate expanded checkpoint tuples and asserts `evaluationStatus ==
"fixture_invalid"` with all metric values unscored. Consider making duplicate
detection impossible to bypass by removing scored duplicate handling from
`_quality_metrics()` or making it a defensive evaluator error.

**Blocks Phase 4 certification:** Yes  
**Blocks V0.8 release readiness:** Yes

### Low

#### L1 - H-gate failure reporting is indirect

**Evidence:** H1-H7 boundary cases exist, but failures are represented primarily
through aggregate-status and safety failures rather than dedicated H-gate result
records.

**Impact:** This is auditable but less precise than ADR-005's ideal failure
recording. It does not independently invalidate the benchmark once higher-priority
safety instrumentation is fixed.

**Required disposition:** Add diagnostic H-gate result details or enrich
aggregate/safety failure messages with the H gate under test.

**Blocks Phase 4 certification:** No  
**Blocks V0.8 release readiness:** No

#### L2 - Non-policy non-overridable decision rejection deserves a dedicated case

**Evidence:** Invalid decision coverage includes blocked policy and keep/unknown
targets. Non-overridable blocker behavior is represented through the policy
blocked path, but not a separate missing-keeper or duplicate-removal blocker
decision case.

**Impact:** Current coverage is acceptable, but a dedicated case would make the
decision-application matrix clearer and more robust.

**Required disposition:** Add a focused decision-stage case targeting a
non-policy non-overridable blocker.

**Blocks Phase 4 certification:** No  
**Blocks V0.8 release readiness:** No

## Release Certification Impact

Phase 4 should not be certified yet. The benchmark is runnable and close, but
the safety and non-vacuity defects affect release-gating claims.

Required fixes before Phase 4 certification:

1. Strengthen `source_report_immutability` and `no_effects` instrumentation.
2. Replace `coverageSubtypes` non-vacuity with scored exact subtype coverage or
   explicit version-specific subtype dispositions.
3. Add fixture-invalid duplicate checkpoint tests and ensure duplicate rejection
   cannot be treated as a scored benchmark failure.
