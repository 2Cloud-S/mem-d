# V0.7 Simulation Architecture Audit

Date: 2026-06-17  
Auditor role: Independent architecture review (pre-implementation)  
Document reviewed: `docs/design/V0.7-SIMULATION-ARCHITECTURE.md`  
Method: Cross-check against ADR-001, ADR-002, V0.6 recommendation architecture, `memd/contracts.py`, `memd/recommendations.py`, `memd/metrics.py`, and `memd/pipeline.py`

**No code, contracts, tests, or implementation files were modified.**

---

## Executive Summary

The V0.7 Simulation Architecture is **well-aligned** with the Mem-D progression established in ADR-001 and the evaluation gate in ADR-002. It correctly positions simulation as a **read-only, in-memory, structural projection layer** between benchmarked recommendations and future workflow/action systems.

The design **does not introduce execution behavior**, **does not duplicate recommendation conflict resolution**, and **does not re-infer lifecycle, evolution, or governance**. Simulation semantics (merge = removable removal, archive = active-store removal with log retention, review/keep = no structural change) are consistent with V0.6 recommendation intent.

Primary weaknesses are **metric definition precision**, **explainability fallbacks when `Recommendation` records are absent**, and **underspecified formulas** for post-simulation compression metrics. These are architectural documentation gaps, not fundamental layer-boundary failures.

**Verdict: PASS WITH LIMITATIONS**

Implementation may proceed after resolving the medium-severity metric and explainability clarifications listed under Recommended Adjustments. No redesign of the simulation layer boundary is required.

## Architecture Strengths

1. **Correct layer placement** — Simulation sits after recommendation generation and evaluation, matching ADR-001 progression (`Diagnostics → Recommendations → Simulation → Workflows → Actions`).

2. **Authoritative input contract** — `MemoryResolution` is the primary simulation driver; `Recommendation` is secondary for roles, grouping, and explainability. This matches V0.6 implementation where every memory receives exactly one resolution (`plan_recommendations` iterates all `memory_ids`).

3. **No upstream duplication** — Architecture explicitly forbids re-inferring lifecycle, re-applying policy, or re-running conflict resolution. Governance actions and trust data are used for context and utilization only.

4. **Structural-only semantics** — Merge does not fuse content; archive does not call external retirement APIs; review/keep are no-ops on store structure. This preserves Mem-D read-only posture per `AGENTS.md`.

5. **Safety model** — Copy-on-simulate, source immutability, no I/O, no elevation of `review` to `merge`/`archive`, and explicit `simulationWarnings` for edge cases are appropriate and testable.

6. **ADR-002 boundary** — Document states simulation measures operational impact assuming resolutions are correct; it does not conflate simulation benchmarks with recommendation accuracy.

7. **Determinism by design** — Fixed handler order, no model calls, no randomness, proposed `simulationId` hash for reproducibility.

8. **Future-oriented outputs** — `simulatedMerges`, `simulatedArchives`, and `simulatedReviewQueue` provide workflow-ready groupings without implying orchestration.

## Findings Ordered By Severity

### High

**None.** No finding blocks architectural acceptance.

### Medium

#### M1 — Duplicate/compression metric definitions misalign with `memd/metrics.py`

The architecture defines:

- `duplicateRecordsBefore` as “count of records in duplicate clusters”

Current metrics compute:

- `duplicate_count = len(duplicate_members) - len(clusters)` (removable estimate, not member count)
- `compressionOpportunity` = `duplicate_percentage` derived from removable count / total

Post-simulation `compressionOpportunityAfter` and `trustedCompressionAfter` are described as “estimated duplicate %” and “trusted removable count / active count” without a normative formula matching `calculate_metrics()`.

**Risk:** Simulation deltas may be **misleading or non-comparable** to `AnalysisReport.metrics` and benchmark baselines (LongMemEval, PERMA).

**Severity rationale:** Metric integrity issue; does not break safety boundaries.

---

#### M2 — Explainability assumes `Recommendation` lookup for all resolutions

V0.6 filters `report.recommendations` to records with evidence (`recommendation_has_evidence`). Implicit `keep` resolutions use `recommendationId` like `rec:implicit:keep:{id}` **without** a corresponding `Recommendation` in `report.recommendations`.

Architecture trace chain requires:

```
MemoryResolution → Recommendation (reason, evidence[])
```

For implicit keeps and some no-op paths, the chain breaks unless simulation falls back to `MemoryResolution` + `validation` evidence directly.

**Risk:** Explainability completeness benchmark (§4) may fail on real `analyze_file` outputs even when simulation is correct.

---

#### M3 — Orphan-merge downgrade is simulation-side reinterpretation

Rule: if `resolvedAction == merge` but no keeper exists in group → warning + treat as `review` (no removal).

This is a **simulation safety override**, not strict consumption of the resolution. It is defensible but should be documented as an explicit simulation guard, not silent alignment with ADR-001 output.

**Risk:** Minor boundary tension with “simulation does not override recommendations”; acceptable if labeled as simulation-only safety semantics.

---

#### M4 — `approvedActionsUtilized` join logic underspecified

Mapping governance actions (cluster-scoped) to per-memory simulated changes is non-trivial:

- One `merge_cluster` action spans multiple memories
- Blocked actions may still produce `review` resolutions with simulated no-op
- Archive actions do not exist as governance action types; archive flows from lifecycle

**Risk:** Utilization metric may be ambiguous or double-counted without a defined join table.

---

### Low

#### L1 — Selective simulation modes (`merge_only`, `archive_only`) affect metric comparability

Modes are useful but `SimulationMetrics` from partial simulation are not comparable to `full` mode. Architecture does not require mode-specific benchmark baselines.

#### L2 — `lifecycleDistributionAfter` definition ambiguous

“Excluding archived/removal IDs from active denominator” does not specify whether removed memories are omitted from state counts entirely or reclassified. Different interpretations yield different distributions.

#### L3 — `simulationId` hash inputs not specified

Determinism claim depends on hashing explicit fields (memories, resolutions, mode). Without a canonical input set, cross-environment reproducibility of `simulationId` is unspecified.

#### L4 — Regression guard may be brittle

“Approved HIGH-trust merges produce non-zero `duplicateReduction` when merge resolutions exist” may not hold on review-dominant exports (e.g. PERMA `user108` posture) where merge resolutions are sparse.

#### L5 — `evidenceRefs` format unspecified

Proposed as `tuple[str, ...]` without convention (`actionId`, `ruleId`, `caseId`, or serialized evidence keys). Cross-report linking may be inconsistent without a standard.

## Layer Boundary Analysis

### A. Simulation does not introduce execution behavior

| Check | Result |
| --- | --- |
| No memory mutation (source) | **Pass** — copy-on-simulate; source `AnalysisReport.memories` immutable |
| No persistence | **Pass** — explicit prohibition; in-memory only |
| No workflow triggering | **Pass** — no queues, schedulers, or automation |
| No action execution | **Pass** — no provider write-back, no content fusion |
| No recommendation generation | **Pass** — consumes `memoryResolutions`; does not call `plan_recommendations` |
| No policy re-evaluation | **Pass** — policy consumed via existing action annotations only |

**Note:** Simulation **does** mutate the in-memory copy (`simulated_store` removals). This is correctly scoped and does not violate Mem-D’s read-only analysis posture.

### B. Simulation consumes recommendations rather than recreating them

| Duplication risk | Result |
| --- | --- |
| Lifecycle re-inference | **Pass** — consumes `memoryLifecycleAssignments` only |
| Evolution re-inference | **Pass** — no evolution audit in simulation path |
| Governance re-planning | **Pass** — `actions` used for context/utilization only |
| Conflict-resolution re-run | **Pass** — uses `MemoryResolution.resolvedAction` as authoritative |

**Caveat (M3):** Orphan-merge guard is simulation-side safety logic, not ADR-001 re-resolution.

### C. Simulation outputs are structural projections only

| Action | Expected semantics | Architecture | Result |
| --- | --- | --- | --- |
| `merge` | Remove removables; no content fusion | Matches | **Pass** |
| `archive` | Remove from active store; preserve in log | Matches | **Pass** |
| `review` | No structural change | Matches | **Pass** |
| `keep` | No structural change | Matches | **Pass** |

## Safety Analysis

| Guarantee | Assessment |
| --- | --- |
| Source immutability | Strong — explicitly required; testable via property tests |
| Copy-on-simulate | Strong — `deep_copy` before handlers |
| No side effects | Strong — no I/O/network in module |
| No elevation (`review` → `merge`/`archive`) | Strong — V0.7 only supports review-as-no-op |
| Deterministic handler order | Strong — merge → archive → classify |
| Ambiguity handling | Adequate — warnings over silent removal |
| Execution authorization | Strong — simulation labeled informational |

**Residual risk:** Consumers misreading `SimulationReport.simulatedMemories` as production state. Mitigation: mandatory “simulated/dry-run” labeling (documented; implementation enforcement still required).

## Metric Analysis

### Computable from current data

| Metric | Computable | Notes |
| --- | --- | --- |
| `memoryCountBefore/After/Delta` | Yes | Direct |
| `memoryReductionPercentage` | Yes | Guard `memoryCountBefore == 0` |
| `duplicateRecordsBefore` | Partial | Definition must align with `duplicateCount` or `duplicate_members` — currently ambiguous (M1) |
| `duplicateRecordsAfter` | Partial | Structural adjustment from source cluster maps is feasible but ≠ re-clustering |
| `compressionOpportunityBefore` | Yes | From `AnalysisReport.metrics` |
| `compressionOpportunityAfter` | Partial | Formula required (M1) |
| `trustedCompressionBefore/After` | Partial | Trusted removable adjustment needs cluster-trust mapping per removed ID |
| `lifecycleDistributionBefore` | Yes | From `memoryLifecycleAssignments` |
| `lifecycleDistributionAfter` | Partial | Denominator/state rules ambiguous (L2) |
| `recommendationUtilizationRate` | Partial | Needs precise `resolutionsApplied` definition |
| `approvedActionsUtilized` | Weak | Join logic underspecified (M4) |
| `unresolvedReviewCount` | Yes | Count `resolvedAction == review` |
| `conflictReviewCount` | Yes | Filter `conflictDetected` |

### Determinism and reproducibility

- Handler order and resolution-driven logic: **deterministic**
- Structural adjustment without re-embedding: **deterministic**
- `simulationId`: **conditionally deterministic** pending canonical hash input spec (L3)

### Double-counting risks

| Area | Risk |
| --- | --- |
| `resolutionsApplied` vs per-group logs | Medium — one merge group removes N memories → N applied resolutions vs 1 group |
| `approvedActionsUtilized` | Medium — cluster actions vs per-memory effects (M4) |
| `suppressedActionCount` | Low — aggregate across resolutions; define dedup rule |
| Merge then archive on same ID | Low — architecture skips with warning |

### Misleading metrics

1. **`compressionOpportunityAfter` without re-clustering** — may **overstate improvement** by removing removables from pre-computed clusters without recomputing cluster integrity. Document must label these as **structural estimates**, not analyze-equivalent metrics.

2. **`duplicateRecordsBefore` using member count** — would not match published benchmark duplicate semantics.

3. **`recommendationUtilizationRate`** — high utilization is not inherently good; architecture correctly treats distribution metrics as diagnostic elsewhere, but utilization could be misread as quality. Label as operational, not correctness.

## Future Compatibility Analysis

### V0.8 Workflows

**Can consume `SimulationReport` without redesign:** **Mostly yes**

Workflows would likely need:

| Present in proposal | Workflow use |
| --- | --- |
| `simulatedReviewQueue` | Human review queues |
| `simulatedMerges` (keeper + removedIds) | Merge work packages |
| `simulatedArchives` | Archive work packages |
| `simulationWarnings` | Escalation / blocked automation |
| `metrics.unresolvedReviewCount` | Queue sizing |
| `simulationMode` | Partial dry-runs |

**Likely extensions (not redesign):**

- `requiresHumanApproval` propagated from `Recommendation` onto simulated events
- Priority / ordering fields for queue scheduling
- Batch identifiers linking multiple `recommendationId` values

No fundamental schema break anticipated if extensions are additive.

### V1.0 Action Systems

**Can consume `SimulationReport` without redesign:** **Partially**

Action systems need execution payloads (survivor content strategy, provider IDs, rollback tokens). Simulation correctly omits these.

`SimulationReport` provides:

- Candidate memory IDs and roles
- Structural impact estimates
- Evidence references for audit

Action layer will require **additional contracts** (execution plans, idempotency keys, provider adapters). That is expected and does not invalidate V0.7 design.

**Risk:** Teams treating `simulatedMemories` as post-execution truth. Architecture mitigates via labeling; action ADR should reference simulation as pre-execution only.

## Benchmarkability Analysis

### Gold fixtures

**Feasible without subjective judgment.**

`simulation_gold.json` can specify:

- Input memories + resolutions (embedded or fixture-built `AnalysisReport`)
- Expected active memory IDs after simulation
- Expected merge/archive logs
- Expected counts and deltas

Structural exact-match tests align with ADR-002 gold-fixture discipline.

### Property tests

Architecture lists appropriate invariants:

- Source immutability
- Idempotency
- Monotonic reduction under merge/archive-only removal
- Review preservation
- Orphan removal logging

All are objectively testable.

### Regression benchmarks

LongMemEval/PERMA guards (inequality/distribution) are **objective but weaker** than gold fixtures. Guard L4 (non-zero duplicate reduction when merges exist) should be conditional on fixture containing merge resolutions.

### Explainability benchmark

“Every simulated removal has `recommendationId` and `evidenceRef`” is testable but must account for M2 fallback paths for implicit resolutions.

**Overall benchmarkability: Good**, with metric formula lock-in required before publishing benchmark numbers.

## Recommended Adjustments

Architecture document only — no implementation prescribed.

1. **Normative metric appendix** — Define `duplicateRecordsBefore/After`, `compressionOpportunityAfter`, and `trustedCompressionAfter` using formulas aligned with `memd/metrics.py`, or explicitly rename metrics to `structuralDuplicateEstimate*` to avoid benchmark confusion (addresses M1).

2. **Explainability fallback rule** — When `Recommendation` is absent for a `recommendationId`, trace via `MemoryResolution` + `validation` lifecycle/evolution evidence. Update explainability section and benchmark rule accordingly (M2).

3. **Document orphan-merge guard** — Classify as `SimulationWarning` semantics, not recommendation override (M3).

4. **Specify `approvedActionsUtilized` join** — e.g. action utilized iff any `sourceActionId` on a simulated event matches an approved `GovernanceAction.actionId` (M4).

5. **Clarify `lifecycleDistributionAfter`** — State whether removed memories are excluded from counts or moved to an `Archived` bucket (L2).

6. **Canonical `simulationId` input list** — Memories, resolutions, mode, policy profile (L3).

7. **Mode-aware benchmarking** — Require `simulationMode` in fixture metadata when comparing metrics (L1).

8. **Soften regression guard L4** — Make conditional on presence of merge resolutions in input (L4).

9. **Standardize `evidenceRefs` encoding** — e.g. `governance_action:{actionId}`, `lifecycle:{caseId}`, `policy:{ruleId}` (L5).

10. **Label post-simulation compression metrics** — “Structural estimate (no re-clustering)” in all report surfaces tied to M1.

## Final Verdict

### **PASS WITH LIMITATIONS**

**Justification:**

The V0.7 Simulation Architecture is **fit for implementation** as the next Mem-D layer. It satisfies the audit requirements:

| Requirement | Status |
| --- | --- |
| A. No execution behavior | Pass |
| B. Consumes recommendations; no upstream duplication | Pass (minor simulation guard caveat) |
| C. Structural projections only | Pass |
| D. Metric integrity | Pass with limitations — formulas need lock-in |
| E. Future workflow/action compatibility | Pass with expected additive extensions |
| F. Explainability traceability | Pass with limitations — implicit-keep fallback needed |
| G. Objective benchmarkability | Pass |

**Limitations** are documentation precision issues (metrics, explainability fallbacks, utilization joins), not layer-boundary or safety failures. Resolve medium findings M1–M4 in the architecture document before implementation benchmarking claims are published.

**Not required:** Redesign of simulation scope, pipeline position, or separation from ADR-002.
