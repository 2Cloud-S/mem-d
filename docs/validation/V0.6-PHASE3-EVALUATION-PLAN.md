# V0.6 Phase 3 — Evaluation Plan

Status: Planning (not started)

Date: 2026-06-16

Scope: recommendation evaluation infrastructure only

---

## Executive Summary

Mem-D V0.6 Phases 1 and 2 delivered a complete recommendation output path. Recommendations (`merge`, `archive`, `review`, `keep`) are generated by `plan_recommendations()`, integrated into `analyze_file()`, and exposed in JSON, Markdown, and terminal reports. Architecture audits for Phase 1 and Phase 2 are complete.

- **ADR-001** defines recommendation **generation**: contracts, precedence, conflict resolution, evidence requirements, and pipeline integration.
- **ADR-002** defines recommendation **evaluation**: quality definition, required metrics, gold-fixture discipline, and the progression gate before simulation, workflows, or action systems.

Phase 3 exists to make recommendation quality **measurable**. Today, gold labels exist and a pass/fail resolution test runs in CI, but Mem-D does not yet produce aggregated accuracy metrics, conflict-resolution rates, or reproducible benchmark artifacts comparable to lifecycle and evolution evaluation.

**Phase 3 is evaluation-only.**

No recommendation behavior changes are planned. Phase 3 does not alter precedence, conflict resolution, trust gating, evidence rules, pipeline ordering, or report rendering. It adds measurement, reporting, and regression gates on top of the existing engine.

## Phase 3 Objectives

### Objective 1 — Create reproducible recommendation evaluation

Deliver a deterministic evaluation runner that loads `recommendation_gold.json`, invokes `plan_recommendations()` per case with the same inputs as unit tests, and produces identical metrics on repeated runs.

### Objective 2 — Measure recommendation quality

Compute and report overall accuracy, per-action accuracy, and conflict-resolution accuracy as defined in ADR-002. Emit human-readable and machine-readable benchmark artifacts.

### Objective 3 — Prevent recommendation regressions

Integrate evaluation into CI. Fail builds when accuracy falls below maintainer-defined thresholds or when any gold resolution mismatches.

### Objective 4 — Establish a progression gate

Document measurable completion criteria that must be satisfied before simulation (v0.7), workflows (v0.8), or action systems (v1.0) are considered. Phase 3 deliverables satisfy ADR-002 gate item 1 (runner exists in CI); threshold policy is a maintainer decision recorded in benchmark documentation.

## Evaluation Unit

**Primary evaluation target: `MemoryResolution`**

Each gold case defines `expectedResolutions` — a map of `memoryId` → expected outcome. The runner compares predicted `MemoryResolution` records from `plan_recommendations()` against these labels.

| Property | Why `MemoryResolution` is primary |
| --- | --- |
| One `resolvedAction` per memory | Matches ADR-001 authoritative output; aligns with workflow queue semantics |
| Deterministic conflict outcomes | `conflictDetected`, `suppressedActions`, `role` are evaluable per memory |
| Stable benchmark unit | Same inputs always produce the same resolution (tested in Phase 1) |
| Coverage without `--include-keep` | Implicit `keep` resolutions are evaluated even when group `Recommendation` records are omitted |

**Secondary artifact: `Recommendation` (group-scoped records)**

Group recommendations remain important for report rendering and evidence display, but Phase 3 quality metrics are computed from `MemoryResolution` unless a future ADR extends scope to group-level precision/recall.

Optional secondary checks (reporting only, not gate metrics):

- Non-empty `evidence` on emitted `Recommendation` records
- `recommendationSummary` counts consistent with resolution distribution

## Metrics

All metrics are computed from gold `expectedResolutions` vs predicted `MemoryResolution` unless noted.

### Overall accuracy

**Definition:** Fraction of memory-level resolutions where `predicted.resolvedAction == expected.resolvedAction`.

**Formula:**

```
overall_accuracy = correct_resolutions / total_resolutions
```

Where `total_resolutions` = sum of memory IDs across all gold cases in `expectedResolutions`.

**Target reporting format:**

```text
Overall accuracy: 0.9524 (20/21 resolutions)
```

JSON:

```json
{
  "overallAccuracy": 0.9524,
  "correctResolutions": 20,
  "totalResolutions": 21
}
```

**CI gate (proposed):** `overall_accuracy == 1.0` on current 14-case fixture (all cases must pass; no partial credit at launch).

### Per-action accuracy

**Definition:** Accuracy stratified by **expected** `resolvedAction` in gold labels.

**Formula (per action `a`):**

```
per_action_accuracy[a] = correct_a / total_a
```

Where `total_a` = resolutions where `expected.resolvedAction == a`.

**Actions:** `merge`, `archive`, `review`, `keep`

**Target reporting format:**

```text
Per-action accuracy:
  merge:   1.0000 (3/3)
  archive: 1.0000 (5/5)
  review:  1.0000 (10/10)
  keep:    1.0000 (3/3)
```

JSON:

```json
{
  "perActionAccuracy": {
    "merge":   { "accuracy": 1.0, "correct": 3, "total": 3 },
    "archive": { "accuracy": 1.0, "correct": 5, "total": 5 },
    "review":  { "accuracy": 1.0, "correct": 10, "total": 10 },
    "keep":    { "accuracy": 1.0, "correct": 3, "total": 3 }
  }
}
```

**Guard metrics (diagnostic, reported alongside):**

| Metric | Formula |
| --- | --- |
| False positive rate (action `a`) | Predicted `a` when gold expected different action / total predicted `a` |
| Miss rate (action `a`) | Gold expected `a` but prediction differed / total gold `a` |

### Conflict resolution accuracy

**Definition:** Fraction of conflict-tagged gold cases where **every** memory in the case matches gold on all required fields.

**Case passes when:**

1. `predicted.resolvedAction == expected.resolvedAction` for every memory in the case
2. `predicted.conflictDetected == expected.conflictDetected` when gold specifies `conflictDetected`
3. `predicted.role == expected.role` when gold specifies `role`
4. `expected.suppressedActions` ⊆ `predicted.suppressedActions` (when gold adds `suppressedActions` labels in a future fixture revision)

**Formula:**

```
conflict_accuracy = passed_conflict_cases / total_conflict_cases
```

**Proposed conflict case set (6 cases):**

- `conflict_archive_merge`
- `conflict_merge_review_blocked`
- `conflict_archive_keep`
- `review_policy_blocked_merge`
- `review_unknown_category` (escalation conflict: review vs keep)
- `review_mixed_lifecycle`

**Target reporting format:**

```text
Conflict resolution accuracy: 1.0000 (6/6 cases)
```

JSON:

```json
{
  "conflictResolutionAccuracy": 1.0,
  "passedConflictCases": 6,
  "totalConflictCases": 6,
  "failures": []
}
```

### Distribution metrics

**Definition:** Counts of resolutions and group recommendations by action category across all gold cases.

**Purpose:** Diagnostic and drift awareness only. **Not quality metrics.**

**Sources:**

- Resolution distribution: count `MemoryResolution.resolvedAction` per action
- Recommendation distribution: `RecommendationSummary` aggregates (mergeCount, archiveCount, reviewCount, keepCount)

**Target reporting format:**

```text
Resolution distribution (diagnostic):
  merge: 3 | archive: 5 | review: 10 | keep: 3

Recommendation record distribution (diagnostic):
  merge: 1 | archive: 0 | review: 4 | keep: 1
```

JSON:

```json
{
  "distribution": {
    "resolutions": { "merge": 3, "archive": 5, "review": 10, "keep": 3 },
    "recommendations": { "merge": 1, "archive": 0, "review": 4, "keep": 1 }
  },
  "diagnosticOnly": true
}
```

| Gap | Priority | Notes |
| --- | --- | --- |
| Keeper tie-breaking (newest timestamp → confidence → ID) | Medium | Architecture Phase 3 item; requires new gold cases when behavior is formalized |

| PERMA-scale export labels | Out of scope | Distribution regression only; no per-memory ground truth |

### Expansion rules (future, post-Phase 3 baseline)

Every new recommendation capability **must** include corresponding gold cases before merge:

1. One positive case per new rule or mapper path
2. One false-positive guard where applicable
3. Conflict case if the rule interacts with existing precedence
4. Labels must match architecture/ADR intent, not ad-hoc test convenience
5. Case `id` must be descriptive and stable (used in failure reports)
6. Disputes over expected outcomes require ADR or architecture update before label change

**Phase 3 constraint:** This plan does not require fixture changes to declare Phase 3 complete. The runner must achieve 100% accuracy on the **existing** 14 cases. Fixture expansion is a separate, ADR-governed activity.

## Evaluation Runner Plan

**Proposed module:** `memd/evaluation/recommendations.py` (or `memd/benchmarks/recommendation_evaluation.py`)

**Proposed CLI entry (optional):** `python -m memd evaluate-recommendations` or `python scripts/run_recommendation_evaluation.py`

### `evaluate_recommendations(...)`

**Purpose:** Load gold fixture, run planner per case, compute metrics, return structured result.

**Inputs:**

| Parameter | Type | Description |
| --- | --- | --- |
| `gold_path` | `Path` | Default: `tests/fixtures/recommendation_gold.json` |
| `include_keep` | `bool` | Per-case override from `includeKeep`; default `False` |
| `conflict_case_ids` | `set[str]` | Optional override for conflict-tagged cases |

**Processing steps:**

1. Load and validate gold fixture shape
2. For each case: build inputs via shared helpers (reuse from `test_recommendations.py` or extract to `memd/evaluation/fixtures.py`)
3. Call `plan_recommendations(memories, clusters, validation, insights=(), actions, include_keep=case.includeKeep)`
4. Index resolutions by `memoryId`
5. Compare against `expectedResolutions`
6. Aggregate overall, per-action, conflict, and distribution metrics
7. Collect failure details (case ID, memory ID, expected vs actual)

**Outputs:**

| Output | Type | Description |
| --- | --- | --- |
| `RecommendationEvaluationResult` | dataclass | Structured metrics + failures |
| JSON artifact | file | `examples/benchmarks/recommendation_evaluation.json` |
| Markdown artifact | file | `examples/benchmarks/recommendation_evaluation.md` |

**Expected metrics in result:**

- `overallAccuracy`, `correctResolutions`, `totalResolutions`
- `perActionAccuracy` (merge, archive, review, keep)
- `conflictResolutionAccuracy`, `passedConflictCases`, `totalConflictCases`
- `distribution` (resolutions + recommendations, diagnostic)
- `failures[]` — `{caseId, memoryId, field, expected, actual}`
- `metadata` — `{fixturePath, caseCount, evaluatedAt, memdVersion}`

## Risks

| Risk | Description | Mitigation |
| --- | --- | --- |
| **Gold fixture bias** | 14 synthetic cases may not represent production memory stores | Document scope clearly; treat as architecture-compliance benchmark, not production quality proof; expand fixtures under ADR-governed rules |
| **Overfitting to fixtures** | Engine tweaks that pass gold but fail on real exports | Keep PERMA/LongMemEval distribution regression notes; require new gold cases for new rules; code review discipline |
| **Benchmark drift** | Runner and `test_recommendations.py` helpers diverge | Extract shared fixture loading into one module; single source of truth for case execution |
| **Metric misinterpretation** | High accuracy on gold mistaken for safe automation | Document that distribution metrics are diagnostic; ADR-002 explicitly excludes user satisfaction and execution outcomes; progression gate requires multiple criteria |
| **False confidence from small datasets** | 21 resolution points is a small sample | Report counts alongside percentages; per-action strata with low `total` flagged in Markdown; plan fixture growth without blocking Phase 3 |
| **Conflict case tagging ambiguity** | Unclear which cases count as conflict scenarios | Define explicit conflict case list in runner config; document in benchmark output |
| **Phase scope creep** | Evaluation phase absorbs keeper selection or CLI work | This plan marks Phase 3 evaluation-only; keeper/CLI deferred to separate work items |

## Final Recommendation

Phase 3 should remain **evaluation-only** and must complete before any simulation, workflow, or action-oriented capabilities are considered.

Mem-D has a working recommendation engine and integrated reporting. What it lacks is the measurable quality layer that lifecycle and evolution already possess. Phase 3 delivers that layer: a reproducible runner, objective metrics, benchmark artifacts, and CI regression protection — without changing what recommendations mean or how they are generated.

Upon Phase 3 completion:

```
ADR-001 (generate) + ADR-002 (validate) + Phase 3 artifacts = cleared to plan v0.7 simulation
```

Until then, recommendation outputs must be treated as **integrated but not yet benchmark-certified** for automation purposes.
