# V0.8 Phase 4 Workflow Evaluation Plan

## 1. Phase Summary

V0.8 Phase 4 implements independent workflow evaluation for:

```python
plan_workflows(...)
apply_workflow_decisions(...)
```

The phase creates a workflow-only gold benchmark that proves workflow planning
architecture compliance before V0.8 release certification. The sole workflow
correctness authority is:

```text
tests/fixtures/workflow_gold.json
```

Phase 4 evaluates planning and decision application only. It does not execute
workflow actions, persist decisions, schedule work, change source memories, or
score external datasets. Recommendation and simulation correctness remain owned
by their separate gold fixtures and metric namespaces.

The implementation must follow the API, runner, and artifact pattern used by
recommendation and simulation evaluation, but it must not copy their simpler
denominator or result-state assumptions. ADR-005 requires stricter checkpoint
identity, non-vacuity, safety, and invalid-result semantics.

## 5. Gold Fixture Design

`tests/fixtures/workflow_gold.json` is the sole source of workflow correctness
truth. It must not import or infer labels from recommendation gold, simulation
gold, prior benchmark artifacts, external validation, or unit-test expectations.

Top-level fixture shape:

```json
{
  "fixtureVersion": "0.8.0",
  "plannerVersion": "2",
  "groundTruthAuthority": "tests/fixtures/workflow_gold.json",
  "description": "V0.8 workflow evaluation architecture-compliance fixture",
  "cases": []
}
```

Each case should contain:

```json
{
  "caseId": "safe_merge",
  "description": "Trusted merge with keeper and removable member",
  "primaryPurpose": "merge operation and ready boundary",
  "input": {
    "builder": "safe_merge",
    "overrides": {}
  },
  "planningOptions": {
    "includeKeep": false
  },
  "expected": {
    "initial": {
      "items": [],
      "reviewRoutes": [],
      "blockers": [],
      "operations": [],
      "steps": [],
      "aggregateStatus": [],
      "summary": [],
      "provenance": []
    },
    "decisionStages": [],
    "identityAssertions": [],
    "safetyProperties": [],
    "diagnostics": {}
  }
}
```

Fixture schema requirements:

- `fixtureVersion`, `plannerVersion`, `groundTruthAuthority`, and `cases` are
  required.
- Every `caseId` is unique and stable.
- Every builder name is from a closed allowlist.
- Every expected quality block expands into explicit checkpoints.
- Safety expectations are separate from quality checkpoints.
- Diagnostic expectations are marked diagnostic-only.
- Partial expected blocks require explicit `matchMode`, checkpoint type, primary
  metric, compared fields, and unexpected-field policy.
- Exact expected blocks compare complete contract-relevant structures.
- Relational assertions cover deterministic identity and fingerprint behavior.
- Literal full hashes are avoided unless a stability reason is recorded.

Expected blocks:

- Initial workflow expectations cover `plan_workflows()` output only.
- Decision stages cover ordered calls to `apply_workflow_decisions()`.
- Post-decision expected states cover operator statuses, aggregate status,
  summary counts, fingerprints, readiness, preserved structures, and atomic
  error behavior.
- Identity assertions compare relations across repeated planning, reordered
  inputs, safety-relevant mutations, and decision stages.

## 6. Fixture Builder Rules

Fixture builders may construct input `AnalysisReport` objects only. They may use
small deterministic source fragments for memories, recommendations, memory
resolutions, simulation reports, policy/action data, and planning options.

Builders must not:

- call `plan_workflows()` to derive expected values;
- call `apply_workflow_decisions()` to derive expected values;
- compute expected aggregate status using `compute_aggregate_status()` or other
  production helpers;
- compute expected routing using production planner logic;
- derive expectations from actual planner output;
- import committed benchmark artifacts as labels;
- import evaluator comparison helpers to synthesize expectations;
- use invalid enum values or `model_construct()` shortcuts to manufacture
  impossible production outputs.

Implementation controls:

- Put builder helpers inside `memd/benchmarks/workflow_evaluation.py` or a
  private internal section of that module for Phase 4 simplicity.
- Keep builder inputs small and readable.
- Emit or reconstruct expanded inputs in failure output.
- Add tests that builder code does not import `plan_workflows()`,
  `apply_workflow_decisions()`, `compute_aggregate_status()`, existing artifact
  JSON, or benchmark result files for label generation.

## 8. Checkpoint Model

Canonical checkpoint tuple:

```json
["caseId", "phase", "checkpointType", "semanticKey"]
```

Display ID grammar for reports:

```text
{caseId}::{phase}::{checkpointType}::{semanticKey}
```

Identity rules:

- The canonical tuple, serialized as canonical JSON, is the true checkpoint ID.
- The display ID is report-only.
- `phase` is `initial` or a declared decision-stage ID.
- `checkpointType` must be from the closed ADR-005 mapping.
- `semanticKey` must be deterministic fixture data and must not depend on
  generated planner IDs alone.
- Reject duplicate canonical tuples.
- Reject duplicate display IDs.
- Reject unknown checkpoint types.
- Reject unmapped checkpoint types.
- Reject multiple primary metric assignments.

Primary metric assignment:

| Checkpoint type | Primary metric |
| --- | --- |
| `item_construction` | `itemConstructionAccuracy` |
| `review_routing` | `reviewRoutingAccuracy` |
| `blocker` | `blockerAccuracy` |
| `operation` | `operationAccuracy` |
| `step_ordering` | `stepOrderingAccuracy` |
| `aggregate_status` | `aggregateStatusAccuracy` |
| `summary_consistency` | `summaryConsistencyAccuracy` |
| `provenance` | `provenanceAccuracy` |
| `decision_transition` | `decisionTransitionAccuracy` |
| `identity` | `identityDeterminismAccuracy` |

Overall accounting:

```text
checkpointCount = count(unique quality checkpoint encoded tuples)
overallWorkflowAccuracy.total == checkpointCount
overallWorkflowAccuracy.passed == count(unique passing quality checkpoint encoded tuples)
```

Safety checks use a separate namespace and never enter quality denominators.
Diagnostics are excluded from quality denominators and `checkpointCount`.

## 9. Quality Metrics

All metrics are release-gating. Every denominator must be non-zero in a valid
fixture. Zero denominator means fixture invalid before scoring.

Failure format:

```json
{
  "kind": "checkpoint_failure",
  "caseId": "...",
  "checkpointId": "...",
  "checkpointTuple": ["...", "...", "...", "..."],
  "checkpointType": "...",
  "primaryMetric": "...",
  "phase": "...",
  "semanticKey": "...",
  "expected": {},
  "actual": {},
  "message": "..."
}
```

| Metric | Expected checkpoint source | Numerator rule | Denominator rule | Comparison policy | Scope |
| --- | --- | --- | --- | --- | --- |
| `overallWorkflowAccuracy` | All quality checkpoints | Unique passing quality checkpoint tuples | All unique quality checkpoint tuples | Inherits assigned checkpoint rule | Initial and decision |
| `itemConstructionAccuracy` | Expected item presence/absence and labeled item-state blocks | Passed item obligations | One semantic item obligation each | Exact fields; partial only if schema-declared and not safety-relevant | Initial and labeled decision |
| `reviewRoutingAccuracy` | Expected review route blocks | Passed route obligations | One route per item | Exact required flag, subtype set, primary queue, full queue refs | Initial and preservation stages |
| `blockerAccuracy` | Expected blocker presence/absence blocks | Passed blocker obligations | One blocker presence or absence each | Exact structured fields; message exact only when labeled | Initial and preservation stages |
| `operationAccuracy` | Expected operation blocks | Passed operation obligations | One operation each | Exact complete operation; no partial credit | Initial and preservation stages |
| `stepOrderingAccuracy` | Expected step order/dependency blocks | Passed step graph obligations | One ordered graph per scope | Exact order, sequence, membership, dependencies | Initial and preservation stages |

## 10. Safety Gates

Safety gates are separate from quality scores. Each property result must include
`passed`, `checksPassed`, `checksTotal`, and `failures`; `checksTotal >= 1` is
mandatory.

| Safety gate | Implementation plan |
| --- | --- |
| Source report immutability | Snapshot canonical `AnalysisReport`, memories, recommendations, simulation, policy/action data before and after planner/decision calls; fail on mutation |
| Deterministic planning | Plan two fresh equivalent reports/options and compare canonical serialized plans and IDs |
| Idempotent repeated planning | Repeat planning in the same process using fresh equivalent source objects; compare outputs |
| Initial never ready | Assert every initial plan has planner status initial, operator statuses none, empty fingerprint, and not `ready_for_execution` |
| Zero structural never ready | Validate empty, all-keep, review-only, all-rejected, and all-deferred cases do not become ready |
| Blocked policy preservation | Validate `POLICY_BLOCKED` blocker, non-overridable behavior, and decision rejection |
| Unresolved policy requires review | Validate unresolved existing action routes to review and differs from missing action |
| Missing source action fail-closed | Validate missing action references produce integrity blocking and no structural readiness |
| Keep never removes | Validate keep items never produce merge/archive removal operations or readiness |
| Merge keeper/removable safety | Validate exactly one keeper, distinct removables, and no archive/review target contamination |
| Integrity suppresses readiness | Validate `INPUT_INTEGRITY` yields `integrity_blocked` and suppresses structural readiness |
| Simulation warnings visible/scoped | Validate recommendation-scoped and memory-scoped warnings affect matching items only; unscoped warning fails closed |
| Structural overlap conflict review | Validate implicated items route to conflict review and withhold structural steps; unrelated control remains unaffected |
| Blocked decisions atomic | Apply invalid mixed decision batch and assert exception plus byte-identical plan |
| Decision preserves steps/operations | Compare steps and operations before and after valid decisions |
| Plan ID stable across decisions | Compare `workflowPlanId` across initial, valid decision, and rejected decision stages |
| Cumulative decision fingerprint | Validate multi-call cumulative state fingerprint and equivalent-state stability |
| No effects | Instrument only the planner/decision call boundary for filesystem writes, network/socket calls, subprocess launches, action execution APIs, persistence/scheduling APIs, and source object mutation |

`no_effects` must not claim more than CI can observe. Runner writes under
`examples/benchmarks/` occur after evaluation returns and are outside the
planner/decision boundary.

## 11. H1-H7 Readiness Boundary Evaluation

Each H gate is evaluated with one positive ready baseline and one negative
control. Boundary violations contribute to quality through `aggregate_status`
checkpoints and to safety through the readiness safety assertions.

| Gate | Input pattern | Expected failure when violated | Assertion category |
| --- | --- | --- | --- |
| H1 | No structural item after resolving all non-structural work | Aggregate is not `ready_for_execution`; failure identifies missing structural item | Quality and safety |
| H2 | One structural item unapproved or ineligible | Aggregate is not ready; failure identifies item and eligibility/approval field | Quality and safety |
| H3 | Approved structural item plus unresolved non-structural actionable item | Aggregate remains `partially_approved` or otherwise not ready | Quality and safety |
| H4 | Approved structural item plus unresolved review item | Aggregate remains `partially_approved`/review-blocked; route failure if review not visible | Quality and safety |
| H5 | Item-level blocker present | Aggregate not ready; blocked item and blocker code recorded | Quality and safety |
| H6 | Plan-level blocker present | Aggregate not ready, usually `integrity_blocked`; plan blocker recorded | Quality and safety |
| H7 | Planning mode `recommendations_only` | Aggregate not ready even if structural-looking work is approved | Quality and safety |

If H5 and H6 cannot be isolated through normal production-valid inputs, document
the closest production-valid fixture pattern and never use invalid contract
objects to force impossible states.

## 12. Decision Application Evaluation

Decision-stage evaluation covers `apply_workflow_decisions()` independently from
initial planning.

Required valid paths:

- approve a merge item;
- approve an archive item;
- acknowledge/approve a review item without creating structural intent;
- reject all actionable items;
- defer all actionable items;
- partially approve structural work while another item remains unresolved;
- transition a fully eligible structural plan to `ready_for_execution`;
- apply decisions across multiple calls and validate cumulative fingerprint.

Required invalid paths:

- unknown target ID;
- decision against keep item;
- decision against blocked item;
- decision against item with non-overridable blocker;
- conflicting duplicate decisions for one target in a batch;
- mixed batch containing an otherwise valid decision plus an invalid target.

Decision assertions:

- invalid batches fail atomically;
- rejected invalid batches leave the input plan byte-identical;
- valid decisions change only operator-derived fields, summary operator counts,
  aggregate status, planner status, and `decisionsFingerprint`;
- `workflowPlanId` remains unchanged;
- steps and operations remain unchanged;
- blockers and evidence remain unchanged;
- fingerprint represents cumulative resulting operator state, not the latest
  batch only.

## 13. Diagnostic Metrics

Diagnostics are non-gating and must be marked `diagnosticOnly: true`.

Planned diagnostics:

- workflow aggregate-status distribution;
- review queue distribution with primary and all-membership views;
- blocker distribution by code and scope;
- planning-mode distribution;
- structural operation distribution;
- review-to-structural ratio with explicit zero-structural handling;
- average workflow items per case;
- average steps per case;
- integrity-blocked case count.

Diagnostics must never change quality denominators, thresholds, failure lists,
or `gatePassed`.

```json
  "evaluationStatus": "scored",
  "fixtureVersion": "0.8.0",
  "plannerVersion": "2",
  "caseCount": 0,

  "failures": [],
  "gatePassed": false,
  "groundTruthAuthority": "tests/fixtures/workflow_gold.json",
  "diagnosticOnly": false

```

Result states:

- `scored`;
- `fixture_invalid`;
- `evaluator_error`;
- `scored_failure`;
- `safety_failure`;
- `artifact_generation_failure`.

Invalid fixture and evaluator error results must include all eleven metric names
with `accuracy: null`, `passed: 0`, and `total: 0`.

`gatePassed` is true only when:

```text
evaluationStatus == scored
all quality metrics == 1.0
all safety properties pass
failures == []
artifact generation, when requested, succeeds
```

## 18. Risks and Controls

| Risk | Control |
| --- | --- |
| Fixture labels accidentally derived from planner output | Builder restrictions, import checks, code review, and tests |
| Denominator inflation | Canonical checkpoint tuples, one primary metric per checkpoint, and set-union overall accounting |
| Reserved blocker codes manufactured with invalid inputs | Required absence checks for `UNSUPPORTED_ACTION` and `STALE_EVIDENCE`; no invalid enum/model shortcuts |
| Composite cases become opaque | Require primary purpose, expanded checkpoint listing, and split cases when failures are hard to localize |
| H5/H6 isolation is difficult | Use closest production-valid pattern and document predicate under test |
| `no_effects` overclaims | Bound instrumentation to planner/decision calls and state proof limits |
| Artifact drift | Exclude timestamps/absolute paths and render both artifacts from one normalized result |
| Test/runner drift | Both call `evaluate_workflows()` and compare normalized result output |
| Planner version changes | Fail on planner-version mismatch; review ADR, fixture, evaluator, and artifacts together |
| Small fixture overconfidence | State architecture-compliance scope in artifacts and release evidence |
| Existing repository baselines confuse results | Explicitly disposition parser fixture and Ruff baselines outside workflow evaluation |

## 19. Completion Criteria

Phase 4 is complete when:

- ADR-005 is Accepted;
- `tests/fixtures/workflow_gold.json` is committed;
- `memd/benchmarks/workflow_evaluation.py` is implemented;
- `tests/test_workflow_evaluation.py` is implemented and passing;

- `overallWorkflowAccuracy.total == checkpointCount`;
- every safety gate is present, non-vacuous, and passing;
- no fixture validation gaps remain;
- `gatePassed == true`;
- `examples/benchmarks/workflow_evaluation.json` is committed;
- `examples/benchmarks/workflow_evaluation.md` is committed;
- `docs/validation/WORKFLOW-BENCHMARK.md` is created;
- `docs/validation/BENCHMARK-EVIDENCE-SUMMARY.md` is updated;
- `docs/validation/V0.8-PHASE4-WORKFLOW-EVALUATION-IMPLEMENTATION.md` is created;
- tests pass except documented unrelated repository baselines;
- independent Phase 4 audit is ready to begin.

## 20. Phase Gate Recommendation

Phase 4 implementation may begin because ADR-005 final verification approved the
architecture with limitations and found no remaining high or medium blockers.

Implementation must not broaden scope into execution, persistence, approval UX,
planner behavior changes, contract changes, or upstream recommendation/simulation
rescoring. The first implementation checkpoint should be fixture schema and
checkpoint validation, not benchmark score generation.
