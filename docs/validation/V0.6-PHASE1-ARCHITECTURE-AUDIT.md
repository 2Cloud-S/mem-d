## Executive Summary

**Overall verdict: Compliant with Phase 1 scope and ADR-001**, with minor gaps against the full V0.6 architecture document. The implementation is read-only, evidence-consuming, and conflict-aware. No memory mutation, execution, or duplicated inference/policy engines were found.

Phase 1 intentionally defers pipeline wiring, report output, and some evidence sources (`insights`, headline/trusted compression metrics). Those deferrals are documented and align with ADR-001.

## Verification Details

### 1. No memory mutation exists — **Pass**

**Evidence:**

- `memd/recommendations.py` constructs frozen Pydantic models (`Recommendation`, `MemoryResolution`) and returns tuples.
- No imports from `memd.parser`, no file I/O, no in-place edits to `MemoryRecord`, `GovernanceAction`, or `validation` payloads.
- No `.model_copy()`, `.update()`, `write`, `delete`, or similar mutation patterns in the module.
- Input `memories` are indexed read-only via `records_by_id`.

**Conclusion:** The recommendation layer is output-only. It does not alter memory stores or upstream analysis artifacts.

---

### 2. No execution logic exists — **Pass**

**Evidence:**

- No functions named or behaving as executors (merge/archive/delete/apply).
- `requiresHumanApproval` is set on recommendations but nothing acts on it.
- `RecommendationAction.MERGE` and `ARCHIVE` are labels, not operations.
- `build_deferred_recommendations` maps `ignore_low_value_issue` to `defer` recommendations only; it does not suppress insights or mutate state.

**Conclusion:** Recommendations are planning artifacts only, consistent with `AGENTS.md` and ADR-001.

---

### 3. No lifecycle inference logic duplicated — **Pass**

**Evidence:**

- No import of `memd.memory_lifecycle` or `infer_memory_lifecycle`.
- Lifecycle data is read from `validation["memoryLifecycle"]["memoryLifecycleAssignments"]` via `lifecycle_assignments_by_id()`.
- Archive eligibility uses pre-computed `lifecycleState` and `confidence` fields only.
- `ARCHIVE_ELIGIBLE_STATES` is a read filter, not an inference ruleset.

**Conclusion:** Lifecycle inference remains upstream. The recommendation layer maps existing assignments to candidates.

---

### 4. No evolution inference logic duplicated — **Pass**

**Evidence:**

- No import of `memd.memory_evolution` or `audit_memory_evolution`.
- Evolution is consulted only for corroboration: `evolution_confidence_for_memory()`, `find_successor()`, and `sourceEvolutionType` on lifecycle assignments.
- No pair detection, regex heuristics, or evolution-type priority logic in `recommendations.py`.

**Conclusion:** Evolution inference is not duplicated.

---

### 5. No governance policy logic duplicated — **Pass**

**Evidence:**

- No import of `memd.policy`, `apply_policy`, or `evaluate_action`.
- Policy is consumed via fields already on `GovernanceAction`: `policyDecision`, `policyRuleId`, `policyExplanation` (explanation not re-derived).
- Recommendation gating (`blocked_merge_review`, `disqualify_merge`, modifiers) **reacts to** policy labels; it does not re-implement conservative/balanced/aggressive rule tables.

**Note:** `MERGE_MIN_CONFIDENCE` (0.80) aligns with balanced policy thresholds. This is recommendation-layer gating, not a second policy engine. Threshold alignment is intentional per architecture.

**Conclusion:** Policy decisions are produced upstream; recommendations read and respect them.

---

### 6. Recommendation confidence derived from existing evidence — **Mostly pass**

**Compliant paths:**

| Action | Confidence source |
| --- | --- |
| `merge` | `min(action.confidence, cluster.trustScore, cluster.averageSimilarity)` via `merge_confidence()` |
| `archive` | `min(lifecycle confidence, evolution case confidence)` when both exist |
| `review` (governance) | `GovernanceAction.confidence` |
| `keep` | `lifecycle confidence * 0.98` (capped) |
| Conflict escalation | `min(archive, merge)` capped at `REVIEW_AMBIGUITY_CAP` |

**Gaps:**

| Gap | Severity | Detail |
| --- | --- | --- |
| Modifier review confidence uses fixed lookup | Low | `review_confidence_for_modifier()` maps modifier strings to constants (0.54, 0.85, etc.) rather than upstream numeric evidence |
| `insights` parameter unused | Low | `plan_recommendations(..., insights)` stored in context but never read; compression insights not attached as evidence |
| `compression_metrics` source never emitted | Low | `SOURCE_PRIORITY` defines `compression_metrics` but no code populates it; ADR PERMA lesson partially addressed via cluster trust, not metrics |
| `default_resolution` fallback confidence `0.5` | Low | Memories with no candidates and no lifecycle assignment get implicit `keep` at fixed 0.5 without evidence tuple |

**Conclusion:** Primary confidence paths are evidence-derived. Secondary paths use architecture-defined constants (acceptable) but do not fully wire compression/insight evidence.

---

### 7. No recommendation bypasses conflict resolution — **Pass**

**Evidence:**

- Every memory in `plan_recommendations` flows: `collect_candidates` → `resolve_memory_conflicts` → `MemoryResolution`.
- Group `Recommendation` records are built **only** from resolutions (`build_group_recommendations`), except `build_deferred_recommendations` which is outside the merge/archive/review/keep taxonomy.
- Archive + merge on the same memory triggers `escalation_review_conflict()` after distinct-action detection.
- Modifiers run before winner selection (`apply_modifiers`).
- Conflict matrix implemented in `conflict_matrix()` for documented pairs.
- Merge candidates disqualified on `policy.blocked` and `cluster.trust.low` before resolution.

**Conclusion:** No path emits a final `merge` or `archive` resolution for competing remediation without passing through conflict logic.

---

### 8. All recommendation outputs are explainable — **Mostly pass**

**Compliant:**

- `Recommendation` requires `reason`, `evidence`, `affected_memories` (contracts enforce structure).
- Merge recommendations include `cluster_trust`, `governance_action`, and `policy` evidence.
- Archive recommendations include `memory_lifecycle` and `memory_evolution` evidence.
- Conflict cases set `conflictDetected`, `suppressedCandidates`, and `suppressedActions`.

**Gaps:**

| Gap | Severity | Detail |
| --- | --- | --- |
| `conflict_review_recommendation` evidence | Medium | Uses `archive_evidence()` only; architecture example includes merge, cluster trust, and `conflict_resolution` evidence together |
| `review_evidence` fallback | Low | When governance evidence missing, falls back to `suppressedActions` string only |
| `category_candidates()` is empty | Info | Taxonomy/unknown handled via modifiers + validation index; function is a no-op stub |
| Deferred recommendations | Info | May have empty `affected_memories` by design |

**Conclusion:** Explainability requirement is met for primary flows. Conflict review records are thinner than the architecture exemplar.

---

### 9. MemoryResolution produces exactly one resolvedAction per memory — **Pass**

**Evidence:**

```python
memory_ids = sorted(records_by_id)
for memory_id in memory_ids:
    ...
    resolutions.append(resolution)
```

- One `MemoryResolution` per input `MemoryRecord`.
- `resolvedAction` is a single `RecommendationAction` enum value.
- `RecommendationSummary.memoryResolutionCount` equals `len(resolutions)`.

**Caveat (not a violation):** Implicit `keep` is assigned even when `include_keep=False` and no lifecycle exists (`default_resolution`, confidence 0.5). Still exactly one action per memory.

**Conclusion:** Hybrid model (one resolution per memory, many group recommendations) matches architecture.

## Findings Summary

### Strengths

1. **Clean separation of concerns** — reads validation and governance outputs; does not re-run inference or policy evaluation.
2. **Deterministic resolution** — sorted memory IDs, stable recommendation IDs, fixed precedence/tie-break tables; tested by `test_plan_recommendations_is_deterministic`.
3. **Safety-biased conflict handling** — archive + merge escalates to `review`; policy-blocked and low-trust merges disqualified.
4. **Contracts match architecture** — models align with V0.6 schema (`RecommendationEvidence`, `suppressedCandidates`, `conflictDetected`).
5. **Gold fixture breadth** — 14 cases covering merge, archive, review, keep, and conflict scenarios.
6. **Regression isolation** — Phase 1 does not alter existing governance pipeline behavior.

### Weaknesses / gaps

| ID | Finding | Severity |
| --- | --- | --- |
| F1 | `insights` accepted but unused | Low |
| F2 | `compression_metrics` / trusted compression not in evidence | Low |
| F3 | `category_candidates()` empty stub | Info |
| F4 | Conflict review evidence incomplete vs architecture example | Medium |
| F5 | Modifier review confidence uses fixed table, not upstream values | Low |
| F6 | Merge `Recommendation` omitted if cluster missing from `clusters_by_id` while resolution stays `merge` | Medium |
| F7 | `default_resolution` assigns `keep` without lifecycle evidence | Low |
| F8 | Gold fixture does not assert `evidence` non-empty on every recommendation | Low |
| F9 | `merge + keeper + archive` on same memory not in gold (only removable + archive) | Low |
| F10 | `insights` / exact-duplicate insight path not wired (architecture gap from Phase 1 scope) | Low |

## Approved Deviations

These deviations are **intentional Phase 1 scope limits**, approved per ADR-001 and `V0.6-PHASE1-IMPLEMENTATION.md`:

| Deviation | Rationale |
| --- | --- |
| Not wired into `pipeline.analyze_file` | ADR Phase 1 deliverable |
| `AnalysisReport` unchanged | Backward compatibility |
| `insights` parameter unused | Deferred to Phase 2; actions carry sufficient governance signal today |
| `compression_metrics` evidence not populated | Merge gated via cluster trust scores instead of headline metrics |
| `category_candidates()` empty | Modifiers + validation indexes implement same behavior |
| `include_keep` defaults false for recommendation records | Architecture default; resolutions still assign implicit `keep` |
| `defer` outside primary taxonomy counts | Matches architecture secondary category |

## Recommended Fixes

Audit-only recommendations for future phases (no code changes in this audit):

### Phase 2 (recommended)

1. **Wire `plan_recommendations` into pipeline** after `apply_policy`, preserving resolution-first flow.
2. **Populate `compression_metrics` evidence** on low-trust/over-cluster review recommendations using `metrics.trustedCompressionOpportunity` from report context.
3. **Use or remove `insights` parameter** — either wire `exact-duplicates` / compression insights or drop from signature until needed.
4. **Enrich `conflict_review_recommendation` evidence** to include merge, archive, cluster trust, and `conflict_resolution` entries per architecture example.

### Phase 3+ (optional)

5. **Guard F6** — if `resolvedAction == merge` but cluster missing, downgrade to `review` or emit diagnostic.
6. **Tighten `default_resolution`** — use `review` or omit resolution when lifecycle unknown instead of `keep` at 0.5.
7. **Gold fixture hardening** — assert non-empty `evidence` on all recommendations; add `merge + keeper + archive` same-memory case.
8. **Replace modifier confidence lookup** with upstream evidence values where available (policy-blocked merge → action confidence cap).

## Conclusion

V0.6 Phase 1 meets the core architecture and ADR-001 requirements for a **read-only, evidence-consuming, conflict-resolving recommendation foundation**. No memory mutation, execution, or duplicated inference/policy logic was identified.

Remaining gaps are **documentation and wiring** (insights, compression evidence, conflict review richness, pipeline integration) rather than fundamental design violations. Phase 1 is **approved to proceed to Phase 2** with the recommended fixes tracked above.
