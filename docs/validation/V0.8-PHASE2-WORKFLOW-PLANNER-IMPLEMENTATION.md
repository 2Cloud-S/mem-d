## 1. Implementation summary

Phase 2 delivers a deterministic, read-only workflow planner in `memd/workflows.py` with:

- Two-pass `plan_workflows()` (normalize/verify/fingerprint → materialize)
- Pure `apply_workflow_decisions()` with cumulative `decisionsFingerprint`
- `compute_aggregate_status()` per ADR-004 precedence rules 1–18
- Canonical SHA-256 identity pipeline and `plannerSignalInputs` fingerprint
- Cross-pass integrity cause accumulation and late `INPUT_INTEGRITY` finalization
- R2/R3 in Pass A; R4 in Pass B after `simulationRefs` assignment

## 4. Two-pass algorithm

**Pass A:** lookup maps, simulation indexes (`simulatedMerges`, `simulatedArchives`, `simulatedReviewQueue`, `simulationWarnings`), audit sets, recommendation normalization/dedup/conflict detection, R2/R3 verification, `sourceAnalysisRef`, `workflowPlanId`.

**Pass B:** item/blocker drafts, policy aggregation, missing-action integrity, review routing, `simulationRefs`, R4 resolution, aggregated `INPUT_INTEGRITY`, frozen items/steps, evidence/summary/aggregate/plan.

Internal drafts use private dataclasses (`PlanningContext`, `ItemDraft`, `BlockerDraft`, `StepDraft`).

---

## 5. Identity implementation

- `model_dump(mode="json", exclude_none=False)` → `json.dumps(sort_keys=True, separators=(",", ":"))` → SHA-256 hex
- `plannerSignalInputs` includes full upstream surfaces, `simulationVerification` payload, conflict-only `integrityCauseKeys`, and derived `auditSets`
- `AnalysisReport.workflowPlan` excluded from fingerprints
- Deterministic IDs: `sourceAnalysisRef`, `workflowPlanId`, `workflowItemId`, `blockerId`, `stepId`, `decisionsFingerprint`

---

## 6. Integrity handling

| Cause | Pass | Effect |
| --- | --- | --- |
| Conflicting duplicate recommendations | A | `integrityCauseKeys` in plan ID; `INPUT_INTEGRITY` |
| R2 simulation ID mismatch | A | `INPUT_INTEGRITY` |
| R3 `analysisReportRef` mismatch | A | `INPUT_INTEGRITY` |
| Missing `sourceActionId` | B | Item `BLOCKED`, `policyDecision=None`, no structural step |
| R4 unresolvable `simulationRefs` | B | Item integrity ref, structural suppression |

Plan-level `INPUT_INTEGRITY` materialized once after all causes collected.

---

## 7. Review and policy routing

- Policy: `blocked > requires_review > approved`; missing actions fail closed
- Structured subtype predicates only (no message parsing)
- `general` fallback when review independently required and no specific subtype
- Safe proposed merge/archive with approved policy and no signals → no review

---

## 8. Step construction

Tiers: Review → Archive → Merge → Retain. Gap-free sequences. Review-before-structural dependencies. Archive+merge memory conflicts withhold structural steps. No `HANDOFF` steps. `STALE_EVIDENCE` not emitted.

---

## 9. Aggregate status

`compute_aggregate_status` implements ADR-004 rules 1–18 including `handoff_ready` H1–H7. Initial plans never `ready_for_execution`.

---

## 10. Decision semantics

- Atomic batch validation; empty decisions return input plan unchanged (same reference)
- Steps/operations immutable across decision application
- Cumulative fingerprint over all item operator states
- Non-overridable blockers and blocked/keep items reject decisions
