# V0.7 Phase 1 — Simulation Layer Architecture & Implementation Audit

Date: 2026-06-17  
Auditor role: Independent post-implementation review (read-only)  
Method: Cross-check implementation against hardened architecture, hardening decisions M1–M4, contracts, tests, and gold fixtures

**No code, contracts, tests, or implementation files were modified.**

---

## Executive Summary

The V0.7 Phase 1 simulation layer **implements the hardened architecture faithfully** for its stated scope: a read-only, deterministic, resolution-driven dry-run that projects hypothetical store outcomes from `AnalysisReport` recommendation outputs without mutating source data or re-running upstream layers.

Core semantics (merge = removable removal with keeper retention, archive = active-store removal with log preservation, review/keep = no structural change), application ordering (merge → archive → review → keep), orphan-merge safety (`ORPHAN_MERGE_NO_KEEPER`), and hardened structural metrics (`estimated*`, `reference*`, recommendation-outcome utilization) are **correctly implemented** in `memd/simulation.py` with **additive contracts** in `memd/contracts.py`.

Safety guarantees hold: `simulate_recommendations` imports only `memd.contracts`, performs no file/network I/O, does not call `plan_recommendations` or any governance/lifecycle/evolution/policy module, and tests confirm input `AnalysisReport` immutability and deterministic output.

Primary gaps are **test coverage depth** (several architecture-listed property invariants and benchmark guards are absent), **partial explainability enrichment** (evolution and categoryQuality fallback paths documented in architecture are not implemented), and **deferred integration surfaces** (pipeline, reports, CLI, simulation modes) which Phase 1 explicitly scoped out.

**Verdict: APPROVED WITH LIMITATIONS**

Phase 2 integration may proceed. Address medium/low findings on explainability enrichment and test gaps before treating simulation as production-complete for operator-facing or benchmark workflows.

**Estimated architecture compliance: 92%**

## Hardening Decision Verification (M1–M4)

### M1 — Structural `estimated*` metrics — **Pass**

Implementation matches normative definitions:

- `removable_estimate(cluster, S)` and `trusted_removable_estimate` use source cluster maps against `S_before` / `S_after`
- `estimatedDuplicateReduction`, `estimatedCompressionGain`, `estimatedTrustedCompressionGain` computed per architecture formulas
- `referenceCompressionOpportunity` / `referenceTrustedCompressionOpportunity` copied from `AnalysisReport.metrics` without implying equivalence
- `METRICS_DISCLAIMER` and `SimulationReport.metricsDisclaimer` present

No post-simulation `calculate_metrics()` or re-clustering observed.

### M2 — Explainability fallback — **Mostly pass**

Implemented:

- `SimulatedExplainability` with `explainabilitySource` (`recommendation` | `resolution_fallback`)
- Normative implicit keep reason templates
- `resolutionSnapshot` on explainability blocks
- Lifecycle assignment refs (`lifecycle_assignment:`, `lifecycle:`)

**Gap:** Architecture normative fallback step 3 also requires enrichment from `memoryEvolutionAudit` and `categoryQuality` samples. `_build_evidence_refs` does not consult these validation keys. See finding **L1**.

### M3 — Orphan-merge safety — **Pass**

Trigger logic, warning code, message text, no `SimulatedMergeGroup`, review-queue downgrade with `orphanMergeDowngrade=True`, and `unresolvedReviewCount` inclusion match hardened semantics. Gold fixture `sim_orphan_merge_no_keeper` and dedicated tests confirm behavior. Input `memoryResolutions` unchanged after simulation.

### M4 — Recommendation-outcome utilization — **Pass**

No governance-action utilization metrics. `recommendationsEligible`, `recommendationsWithStructuralEffect`, and `recommendationOutcomeUtilizationRate` implemented from resolution sets and simulated event logs. `recommendationUtilizationRate` is per-memory removal count / total resolutions per architecture.

## Findings Ordered By Severity

### High

**None.** No finding blocks approval for Phase 1 scope or Phase 2 integration planning.

---

### Medium

#### M-1 — Explainability fallback enrichment incomplete (M2 residual)

**Severity:** Medium  
**Location:** `memd/simulation.py` → `_build_evidence_refs`

Architecture requires fallback enrichment from `validation.memoryEvolutionAudit` and `validation.categoryQuality` (unknown/conflict samples). Implementation enriches lifecycle assignments, cluster trust, governance actions, and recommendation evidence only.

**Impact:** Complex review/archive cases may lack evolution or taxonomy `evidenceRefs` that operators expect from the hardened explainability spec. Does not affect structural simulation correctness.

**Recommendation:** Add evolution `caseId` and category-quality refs in Phase 2 explainability hardening; extend gold fixtures with evolution/category samples.

---

#### M-2 — Architecture property invariants not fully tested

**Severity:** Medium  
**Location:** `tests/test_simulation.py`

Architecture benchmark strategy lists property tests not present in CI:

| Property | Status |
| --- | --- |
| Monotonic reduction (`memoryCountAfter <= memoryCountBefore` when only merge/archive remove) | Missing |
| No orphan removal (every removed ID appears in merge or archive log) | Missing |
| Simulated memories are distinct copies (not aliased to source objects) | Missing |

Existing tests cover immutability, determinism, review/keep preservation, and orphan guard. Gap increases regression risk for edge-case removal accounting.

**Recommendation:** Add parametrized property tests in Phase 2 before benchmark runner work.

---

#### M-3 — `DUPLICATE_REMOVAL_SKIPPED` lacks positive gold coverage

**Severity:** Medium  
**Location:** `tests/fixtures/simulation_gold.json`, `tests/test_simulation.py`

Implementation emits `DUPLICATE_REMOVAL_SKIPPED` when archive targets a memory already removed by merge (lines 139–148). `test_duplicate_removal_warning_when_already_removed` only asserts **absence** of warnings on the mixed case (merge and archive target different IDs). No fixture forces archive-after-merge on the same `memoryId`.

**Impact:** Warning path is implemented but unverified by tests.

**Recommendation:** Add gold case where same memory has merge-removable and archive resolutions, or archive follows merge removal of same ID.

---

### Low

#### L-1 — `simulationMode` fixed to `full`; modes not API-exposed (L1 residual)

**Severity:** Low  
**Location:** `memd/simulation.py` line 33

Architecture documents `full`, `merge_only`, `archive_only` with metric comparability rules. Implementation hardcodes `simulation_mode = "full"` with internal branching hooks for other modes but no public parameter. Documented in Phase 1 implementation report as intentional deferral.

**Impact:** Mode-specific simulation and cross-mode benchmark comparison blocked until API extension.

**Recommendation:** Phase 2 or 4: `simulate_recommendations(report, mode="full")`.

---

#### L-2 — Archive handler does not gate on `role == archive_candidate`

**Severity:** Low  
**Location:** `memd/simulation.py` archive loop (lines 133–169)

Architecture role table specifies `archive_candidate` removal. Implementation keys solely on `resolvedAction == ARCHIVE`. In practice ADR-001 assigns `archive_candidate` role for archive resolutions; mismatch risk is theoretical unless malformed resolutions are injected.

**Recommendation:** Optional defensive role check in Phase 2; low priority.

---

#### L-3 — `analysisReportRef` unused

**Severity:** Low  
**Location:** `SimulationReport.analysisReportRef` defaults to `""`

Architecture lists provenance field. Not populated in Phase 1. No functional impact until report linking is needed.

---

#### L-4 — No LongMemEval/PERMA simulation regression guards

**Severity:** Low (scope deferral)  
**Location:** Benchmark strategy section

Architecture progression gate lists structural smoke guards on benchmark exports. Phase 1 delivers gold-fixture tests only. Consistent with implementation report scope.

**Recommendation:** Phase 3 benchmark runner with documented structural inequalities.

---

#### L-5 — Implicit keep explainability not asserted in tests

**Severity:** Low  
**Location:** `tests/test_simulation.py`

`test_explainability_fallback_on_implicit_keep` verifies resolution action and `unresolvedReviewCount` but does not assert `explainabilitySource == "resolution_fallback"` or reason template text. Merge/archive explainability refs are tested; keep fallback completeness is inferred not verified.

---

#### L-6 — Keeper tie-breaking undocumented

**Severity:** Low  
**Location:** `memd/simulation.py` line 94

Multiple active keepers in one merge group resolve via `sorted(keeper_ids)[0]`. Deterministic and safe; not specified in architecture. Acceptable implementation choice.

---

#### L-7 — Duplicate `memoryId` in resolutions silently overwritten

**Severity:** Low  
**Location:** `resolutions_by_id` dict construction (line 37)

If malformed input contains duplicate `memoryId` entries, last resolution wins. ADR-001 guarantees one resolution per memory in normal pipeline output.

## Detailed Verification Notes

### Safety verification

| Check | Evidence |
| --- | --- |
| Source memories never mutated | `_copy_memory(record.model_copy(deep=True))`; keeper content equality test |
| `AnalysisReport` unchanged | `test_source_report_unchanged_after_simulation` compares `model_dump()` and object identity |
| No recommendation regeneration | `simulation.py` imports only `memd.contracts`; no `recommendations` module |
| No upstream re-evaluation | No imports of `policy`, `memory_lifecycle`, `memory_evolution`, `actions`, `pipeline` |
| No persistence / workflow | No `open`, `write`, `Path`, HTTP, or queue usage in module |

### Simulation semantics verification

**Merge:** Groups by `recommendationId` from merge resolutions; removes sorted removables; keeper content unchanged; `SimulatedMergeGroup` logged with trust context from source clusters.

**Archive:** Removes from `active_store`; preserves `archivedRecord` deep copy in log; captures lifecycle state from pre-computed assignments.

**Review:** `sim_conflict_archive_merge` produces queue entry for `cm_old` with `conflictDetected=True`; store size unchanged. `unresolvedReviewCount` includes review resolutions, orphan downgrades, and `conflictDetected` flags per formula.

**Keep:** Implicit keeps (`rec:implicit:keep:*`) produce no queue entries and no removals — consistent with architecture.

**Ordering:** `sim_mixed_recommendation_set` confirms merge removable (`mx_a`) and archive (`mx_old`) both removed while keeper (`mx_b`) and active successor (`mx_new`) remain.

### Metrics audit

Formulas in `_compute_metrics` align with hardened architecture:

- `memoryCountDelta`, `memoryReductionPercentage` — correct
- `estimatedDuplicateReduction` via cluster removable slots — correct
- `estimatedCompressionGain` / trusted variants — structural percentage delta — correct
- `lifecycleDistributionAfter` excludes removed IDs (verified by `test_lifecycle_distribution_excludes_removed_memories`)
- `archivedByLifecycleState` from simulation log — correct
- Utilization rates — correct definitions; rounded to 4 decimal places

Metrics are not presented as benchmark-equivalent; disclaimer enforced.

### Contract compliance

`SimulationMetrics`, `SimulationReport`, and event models match hardened proposed schemas in architecture Output Contracts section. `explainability` nested on event types (not top-level `evidenceRefs` on merge groups) matches implemented contracts — minor diagram drift in architecture merge representation only; contracts are authoritative.

`Recommendation`, `MemoryResolution`, and `RecommendationSummary` definitions in `memd/contracts.py` were not altered by simulation model addition (additive append only).

**Determinism:** `simulationId` hashes canonical JSON of memories, resolutions, mode, and policy profile per L3. Repeated simulation on same report yields identical `model_dump()` — verified.

## Recommended Phase 2 Scope

1. **Pipeline integration** — optional `simulate=True` on `analyze_file()` returning or attaching `SimulationReport`
2. **Report serialization** — JSON/Markdown/terminal with mandatory structural-estimate disclaimer
3. **CLI** — `memd analyze --simulate` or dedicated simulate command
4. **Test hardening** — monotonic reduction, no-orphan-removal, copy isolation, `DUPLICATE_REMOVAL_SKIPPED` gold case, implicit keep `explainabilitySource` assertion
5. **Explainability enrichment** — `memoryEvolutionAudit` and `categoryQuality` refs in fallback path
6. **Simulation mode API** — expose `merge_only` / `archive_only` when benchmark needs arise

Defer to Phase 3: LongMemEval/PERMA regression runner and aggregated benchmark artifacts.

## Explicit Verdict

### **APPROVED WITH LIMITATIONS**

The V0.7 Phase 1 simulation layer correctly implements the hardened architecture's core contract: read-only, resolution-driven, structurally estimated dry-run simulation downstream of recommendations only. M1, M3, and M4 hardening decisions are fully reflected. M2 is substantially implemented with enrichment gaps. Safety invariants and primary semantics are sound and tested.

Limitations are confined to partial explainability enrichment, incomplete architecture-listed property tests, undeferred integration surfaces (expected), and absence of benchmark regression guards (documented deferral). None require architectural redesign.
