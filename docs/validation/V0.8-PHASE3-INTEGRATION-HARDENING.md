# V0.8 Phase 3.1 Integration Test Hardening

## Executive Summary

Phase 3.1 hardening resolves the Phase 3 integration audit test findings without changing production code. The hardening modifies only workflow integration and CLI tests, adds deterministic coverage for structured workflow-step operations, expands Markdown truncation boundary coverage, adds CLI Markdown verification, and directly verifies renderer immutability.

No planner, pipeline, contract, reporting, fixture, benchmark, configuration, or audit-report files were modified.

## M-TEST-1 Resolution

`test_json_structured_operations_preserved` no longer depends on the standard memory fixture accidentally producing merge steps. It now builds a valid synthetic `WorkflowPlan` with deterministic structured operations and contains no `pytest.skip()` or conditional skip.

The test asserts:

```text
workflow_plan_to_dict(plan) == plan.model_dump(mode="json", exclude_none=False)
```

This closes the audit-blocking structured-operation coverage gap.

## Structured Operation Cases Covered

The synthetic plan covers:

- Merge: `stepType == "merge"`, non-empty `keeperId`, `removableIds`, `recommendationIds`, and empty archive/review targets.
- Archive: `stepType == "archive"`, non-empty `archiveTargetIds`, and empty keeper/removable/review targets.
- Review: `stepType == "review"`, non-empty `reviewTargetIds`, and empty removal/archive targets.
- Retain: `stepType == "retain"` and no removal, archive, or review targets.

## Truncation Boundary Coverage

Markdown truncation now has exact `19`, `20`, and `21` entry boundary coverage for:

- workflow items
- workflow steps
- overridable blockers
- evidence validation references

For `19` and `20` entries, tests assert every entry is visible and no truncation notice appears. For `21` entries, tests assert exactly the first `20` entries are visible and the omitted count is `1`.

The current renderer emits:

```text
... 1 additional entries omitted; use JSON output for the complete plan.
```

The tests assert that existing renderer text. The phrase is not grammatically singular, but it is the accepted current renderer behavior and no production renderer change was authorized for this hardening pass.

Additional coverage verifies:

- non-overridable blockers remain visible beyond the normal display limit
- integrity validation references remain visible beyond the normal display limit
- JSON serialization remains complete and untruncated for the underlying plan
- visible ordering is deterministic

## CLI Markdown Coverage

A real CLI Markdown-output test now invokes the existing `analyze --format markdown` path through `CliRunner`.

It verifies:

- `# Mem-D Analysis Report`
- `## Simulation Summary`
- `## Workflow Plan`
- Workflow Plan appears after Simulation
- `## Compression Explanation` follows Workflow Plan
- planning-only disclaimer appears
- no approval-item prompt appears
- no interactive control text appears
- no execution claim appears in the workflow section

No CLI implementation or flags were changed.

## Renderer Immutability Coverage

Direct before/after serialization checks were added for:

- `report_to_dict(report)`
- `render_markdown(report)`
- `render_terminal(report)`

Each test captures:

```text
before = report.model_dump(mode="json")
before_plan = report.workflowPlan.model_dump(mode="json")
```

and verifies both the `AnalysisReport` and attached `WorkflowPlan` serialize identically after rendering. Terminal immutability uses the existing captured Rich `Console(record=True)` pattern.
