# V0.8 Phase 4 Workflow Evaluation Hardening

Date: 2026-06-23

## Executive Summary

Phase 4 workflow evaluation was hardened after the independent implementation
audit reported one high finding, two medium findings, and two low findings.

The fixes are scoped to the Phase 4 evaluation layer, fixture, tests, benchmark
artifacts, and validation documentation. No workflow planner, workflow
contracts, pipeline, reports, recommendations, simulation, policy, or actions
files were modified.

## Audit Findings Addressed

| Finding | Disposition |
| --- | --- |
| H1 safety gates are not certification-grade | Resolved |
| M1 review subtype non-vacuity can be satisfied by non-scored metadata | Resolved |
| M2 duplicate checkpoint rejection is not consistently proven as pre-scoring | Resolved |

| L2 non-policy non-overridable decision rejection needs a dedicated case | Resolved |

## H1 Safety Instrumentation Fixes

The evaluator now records independent before/after snapshots at the actual
evaluation boundaries:

- before and after `plan_workflows(report, ...)`;
- before and after each `apply_workflow_decisions(plan, decisions)` call.

The `source_report_immutability` safety gate now fails if either the input
`AnalysisReport` or a decision-stage input `WorkflowPlan` is mutated.

The bounded `no_effects` guard now blocks CI-observable effects during pure
evaluator calls:

- `socket.socket`;
- `socket.create_connection`;
- `subprocess.Popen`;
- `Path.write_text`;
- `Path.write_bytes`;
- `open()` in write, append, create, or update modes.

Read-only fixture loading remains allowed. The guard replays every planning
case and every decision stage, yielding `48/48` no-effects checks in the
regenerated artifact. Benchmark artifact writes remain outside
`evaluate_workflows()` and are performed only by `scripts/run_workflow_evaluation.py`.

Project-level action execution, persistence, and scheduling APIs were not added
by Phase 4. No such execution path is invoked by the evaluator.

## M1 Review Subtype Non-Vacuity Fixes

Fixture validation no longer counts `coverageSubtypes`. Review subtype
non-vacuity is satisfied only by scored `review_routing` checkpoints.

Fixture changes:

- `simulation_safety` is scored through `scoped_simulation_warning`;
- `lifecycle` is scored through `review_subtype_matrix_lifecycle_plain`;
- `lifecycle_alternate`, `lifecycle_mixed`, and `low_trust` remain scored
  through `review_subtype_matrix_lifecycle_lowtrust`;
- `orphan_merge_downgrade` remains scored through
  `orphan_merge_missing_keeper`;
- `policy_blocked` is explicitly dispositioned as reserved/not emitted by
  planner version `"2"` and is not counted as scored output.

Tests now prove that metadata-only subtype coverage does not satisfy
non-vacuity and that missing scored subtype coverage returns
`evaluationStatus == "fixture_invalid"` before scoring.

## M2 Duplicate Checkpoint Pre-Scoring Fixes

Duplicate checkpoint tuple and rendered checkpoint ID collision behavior is now
tested through full `evaluate_workflows()` invalid fixtures.

Expected invalid result state:

```text
evaluationStatus: fixture_invalid
gatePassed: false
checkpointCount: 0
quality metric accuracies: null
```

Tests prove both duplicate canonical checkpoint tuples and duplicate rendered
checkpoint IDs are rejected during fixture validation before scoring.

## L1 H1-H7 Failure Reporting Fixes

Aggregate-status checkpoint failures for cases with `expected.hGates` now name
the gate under test. Example shape:

```text
H2 readiness boundary mismatch: aggregate status mismatch
```

A focused test mutates the H2 expectation and verifies the resulting failure
message identifies H2.

## L2 Non-Policy Non-Overridable Decision Case

The fixture now includes a dedicated non-policy non-overridable blocker decision
stage on `orphan_merge_missing_keeper`.

The case attempts to approve `rec:orphan`, which is blocked by non-policy
integrity blockers:

- `MISSING_KEEPER`;
- `ORPHAN_MERGE_NO_KEEPER`.

The expected outcome is an atomic decision rejection with the plan unchanged.
This case contributes to decision-transition coverage and the
`blocked_decisions_atomic` safety gate without relying on policy-blocked
behavior.

## Remaining Limitations

- The `no_effects` guard is bounded to CI-observable Python-level effects. It
  does not claim OS-level sandbox enforcement.
- `policy_blocked` remains a reserved review subtype for planner version `"2"`;
  the planner maps it to a queue but does not emit it from production-valid
  blocked policy cases.
- The full suite still has the known unrelated missing parser fixture baseline.
