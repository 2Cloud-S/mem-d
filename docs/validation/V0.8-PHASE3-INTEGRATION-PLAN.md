## Phase 3 Scope

Phase 3 integrates the verified read-only workflow planner (`memd/workflows.py`, planner version `"2"`) into the standard Mem-D analysis pipeline and reporting surfaces.

### In scope

1. Invoke `plan_workflows()` exactly once after simulation.
2. Attach the initial `WorkflowPlan` to the immutable `AnalysisReport` via `model_copy`.
3. Expose workflow planning in JSON, Markdown, and terminal output.
4. Preserve all existing recommendation and simulation behavior.
5. Read-only planning only — no approval UX, persistence, or execution.

### Out of scope

- `apply_workflow_decisions()` from the pipeline or CLI
- Workflow gold fixtures and Phase 4 evaluation runners
- Contract or planner behavior changes
- Interactive approval controls, persistence, execution

### Phase 3 is

**Workflow pipeline and reporting integration** (ADR-004 roadmap item 3). It is **not** simulation evaluation integration (ADR-003 / V0.7 Phase 3).

## Pipeline Integration Design

### Required pipeline order

```text
Diagnostics
→ Governance
→ Policy
→ Recommendations
→ Simulation
→ Workflow Planning
→ Reporting
```

### Normative `analyze_file()` change

```python
from memd.workflows import plan_workflows

report = AnalysisReport(...)
report = report.model_copy(
    update={"simulationReport": simulate_recommendations(report)}
)
return report.model_copy(
    update={"workflowPlan": plan_workflows(report)}
)
```

### Pipeline invariants

| Rule | Requirement |
| --- | --- |
| Planner input | Report passed to `plan_workflows()` has `simulationReport` attached and `workflowPlan is None` |
| Invocation count | `plan_workflows()` called **exactly once** per successful `analyze_file()` |
| Options | Default `PlanningOptions(includeKeep=False)` via planner default (`options=None`) |
| Decisions | `apply_workflow_decisions()` **never** called from pipeline or CLI |
| Rerun | Recommendation and simulation stages **not** rerun after planning |
| Fallback | No fabricated fallback plan on planner failure |
| Immutability | `plan_workflows()` enforces input report immutability internally |

### Pipeline API decisions

| Decision | Phase 3 choice |
| --- | --- |
| Enabled by default | **Yes** — every successful `analyze_file()` attaches a plan |
| Disable flag | **No** — not added unless implementation discovers a concrete regression |
| CLI approval | **No** |

## Failure Behavior

Planner integrity findings are **data attached to the report**, not pipeline exceptions.

| Scenario | Pipeline behavior |
| --- | --- |
| Empty recommendations | Success; attach plan (`aggregateStatus=empty`) |
| All-keep resolutions | Success; attach plan (`aggregateStatus=all_keep`) |
| Integrity-blocked plan | Success; attach plan unchanged |
| Review / structural plans | Success; attach initial plan |
| Unexpected planner exception | **Propagate**; no partial report returned |
| Upstream stage failure | Existing behavior; planner not reached |

### Normative failure rules

- Planner integrity results attach normally — never converted to generic warnings.
- Unexpected `RuntimeError`, `ValidationError`, or contract failures propagate unchanged.
- No retry, persist, execute, or partial attach after exception.
- No catch-and-swallow of planner programming errors.

Integrity-blocked pipeline behavior is tested via **monkeypatched** `plan_workflows` (see Test Matrix); real integrity logic remains in `tests/test_workflows.py`.

## JSON Reporting

### Null/absent policy (normative)

```text
report_to_dict() omits workflowPlan when None.
```

When `report.workflowPlan is not None`:

```python
payload["workflowPlan"] = workflow_plan_to_dict(report.workflowPlan)
```

This matches `tests/test_workflow_contracts.py::test_report_output_unchanged_when_workflow_plan_none`.

### Exact serialization rule

**One boundary helper only.** No parallel manual serializers for workflow child contracts.

```python
def workflow_plan_to_dict(plan: WorkflowPlan) -> dict[str, object]:
    return plan.model_dump(mode="json", exclude_none=False)
```

Requirements:

- Output **must equal** `plan.model_dump(mode="json", exclude_none=False)` byte-for-byte in JSON structure (test-enforced).
- Enum values and enum-keyed summary dictionaries serialize as JSON strings (Pydantic `mode="json"` behavior).
- Tuples become JSON arrays.
- `WorkflowStep.operation` remains nested structured object.
- All `WorkflowPlan` contract fields included.
- `plannerSignalInputs` **must not appear** — it is not a contract field.

Child helpers (`workflow_item_to_dict`, etc.) are **prohibited** unless an audited incompatibility is discovered before implementation and documented with equality tests against `model_dump` for every helper.

### Target JSON example (contract fields only)

When present:

```json
{
  "workflowPlan": {
    "workflowPlanId": "<64-char hex>",
    "sourceAnalysisRef": "<64-char hex>",
    "simulationId": "<64-char hex or \"\">",
    "policyProfile": "balanced",
    "plannerStatus": "initial",
    "aggregateStatus": "proposed",
    "items": [],
    "steps": [],
    "summary": {
      "totalItems": 0,
      "itemsByAction": {},
      "itemsByPlannerStatus": {},
      "itemsByOperatorStatus": {},
      "itemsByRecommendationPriority": {},
      "reviewQueueCounts": {},
      "blockerCount": 0,
      "keepCount": 0,
      "estimatedStructuralDelta": 0
    },
    "reviewQueues": [],
    "blockers": [],
    "evidence": {
      "recommendationIds": [],
      "memoryIds": [],
      "actionIds": [],
      "insightIds": [],
      "simulationEventIds": [],
      "validationRefs": [],

    },

    "planningOptions": { "includeKeep": false },
    "plannerVersion": "2",
    "metricsDisclaimer": "",

```

Nested item/step/blocker/evidence objects use the same `model_dump(mode="json", exclude_none=False)` shape as defined in Phase 1 contracts.

### Heading hierarchy correction (required)

**Normative change in `render_simulation_summary_markdown()`:**

| Current | Phase 3 |
| --- | --- |

| `## Recommendation Outcomes` | `### Recommendation Outcomes` |
| `## Structural Estimates` | `### Structural Estimates` |

```text
# Mem-D Analysis Report
```

| Impact | Detail |

### Section placement in `render_markdown()`

```python
lines.extend(render_recommendation_summary_markdown(report))
if report.simulationReport is not None:
    lines.extend(render_simulation_summary_markdown(report))
if report.workflowPlan is not None:
    lines.extend(render_workflow_plan_markdown(report))
lines.extend(["## Compression Explanation", ...])
```

When `workflowPlan is None`, omit the Workflow Plan section entirely (not reachable from standard pipeline after Phase 3).

### Data sources (normative)

Render **exclusively** from structured contract fields:

| Source | Use |
| --- | --- |
| `WorkflowPlan.summary` | Counts, queues, delta, keep count |
| `WorkflowItem` | Item lines (action, planner/operator status, queues, refs) |
| `WorkflowStep.operation` | Keeper, removable, archive, review targets |
| `WorkflowBlocker` | Code, message display, overridable flag |
| `WorkflowEvidence` | IDs, validationRefs, warnings |

**Never parse** step descriptions, blocker messages, or recommendation reasons to derive structure. Messages may be displayed verbatim only.

For unscoped duplicate-removal limitation: render existing `INPUT_INTEGRITY` blocker and `integrity:duplicate_removal_unscoped:*` validation refs. **Do not claim** `evidence.warnings` is complete.

### Markdown truncation limits (deterministic)

JSON output remains **complete and untruncated**. Markdown applies these limits in **plan order** (planner sequence / step.sequence):

| Collection | Limit | Sort order |
| --- | --- | --- |
| Steps | First **20** | `step.sequence` ascending |
| Items | First **20** | `queueRank` ascending, then `orderingKey` |
| Review queues | **All** | `reviewQueues` tuple order; counts from `summary.reviewQueueCounts` |
| Blockers — non-overridable | **All** | Plan blocker order |
| Blockers — overridable | First **20** | Plan blocker order after non-overridable |
| Evidence refs — `validationRefs` | First **20** | Sorted tuple order |
| Evidence refs — `simulationEventIds` (per step/item display) | First **20** per parent | Tuple order |
| Evidence refs — `evidenceRefs` (per item) | First **20** per item | Tuple order |

**Integrity rule:** All `INPUT_INTEGRITY` blockers and all `integrity:*` validation refs are shown in full regardless of limits. Safety-critical integrity information is never silently omitted.

**Truncation notice** (required whenever any collection is truncated):

```text
... N additional entries omitted; use JSON output for the complete plan.
```

Replace `N` with exact omitted count.

### Markdown content requirements

- Header: `## Workflow Plan`
- Disclaimer: *Planning only — no memory changes performed. Operator decisions are not applied by Mem-D analysis.*
- Summary: version, mode, aggregate status, planner status (`initial`), policy profile, counts
- Steps: operation fields from `step.operation` only
- Empty/all-keep: concise valid section (never hide workflow layer)

### Markdown heading-order tests (exact)

In `tests/test_workflow_integration.py`:


## Terminal Reporting

### Placement

After `render_simulation_terminal()` in `render_terminal()`, call `render_workflow_terminal(report, console)`.

### Fields (normative)

| Field | Rule |
| --- | --- |
| Header | `[bold]Workflow Plan:[/bold]` |
| Aggregate status | `plan.aggregateStatus.value` |
| Planner state | **`Planner state: initial`** (always for pipeline output) |
| Item count | `plan.summary.totalItems` |
| Review-required count | Count items where `reviewRequirement.required` is true |
| Blocker count | `plan.summary.blockerCount` |
| Structural delta | `plan.summary.estimatedStructuralDelta` |
| Ready-for-execution | **Omit entirely** — initial pipeline plans can never be `ready_for_execution` |
| Planning notice | **`Planning only — operator decisions have not been applied.`** |

### Queue display

- Sort: count **descending**, then queue ID **ascending**
- Show maximum **3** queues
- Format: `review:general (2)`

### Blocker display

- **`INPUT_INTEGRITY` always shown first** (before other blockers)
- Then other blockers in plan order
- Maximum **3** blocker messages total (after integrity priority)
- If more omitted: `... N additional blockers omitted; use JSON output for the complete plan.`

### Terminal constraints

- No item-level dump
- No interactive prompt or approval text
- No execution language ("applied", "executed", "will merge")
- Use existing Rich conventions (`[bold]`, `[red]`/`[yellow]` for integrity-blocked)
- Integrity-blocked: prominent line before blocker list: `[bold red]Workflow integrity blocked[/] — review blockers before manual action.`
