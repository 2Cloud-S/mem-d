# V0.8 ADR-005 Workflow Evaluation Final Verification

Date: 2026-06-23
Type: Focused final verification after ADR hardening
Scope: ADR-005, prior ADR-005 audit, ADR-004, workflow contracts, and workflow planner

## Executive Summary

The hardened ADR-005 resolves the previous architecture-audit blockers. The
document now provides normative coverage requirements, exact denominator
construction for all eleven quality metrics, canonical checkpoint accounting,
non-vacuity rules, bounded safety instrumentation, H1-H7 readiness boundaries,
fixture-builder restrictions, explicit result states, and deterministic artifact
semantics.

No high or medium findings remain. The remaining limitations are implementation
discipline items: Phase 4 must faithfully implement the now-explicit ADR rules
without weakening them in fixture builders, evaluator helpers, or artifact
rendering.

Final verdict:

```text
APPROVED WITH LIMITATIONS
```

Estimated compliance: `97%`

ADR-005 may become Accepted after normal project approval. Phase 4 planning and
implementation may begin from this architecture. This verification does not
change ADR-005 status and does not implement Phase 4.

## Checkpoint Accounting Verification

ADR-005 now defines:

- canonical checkpoint identity as a canonical JSON array of exactly four
  strings: `[caseId, phase, checkpointType, semanticKey]`;
- human-readable display ID construction as report-only;
- component rules for `caseId`, `phase`, `checkpointType`, and `semanticKey`;
- rejection of duplicate canonical tuples;
- rejection of duplicate rendered IDs;
- rejection of unknown checkpoint types;
- rejection of unmapped checkpoint types;
- rejection of multiple primary metric assignments;
- one closed `checkpointType -> primaryMetric` mapping;
- `checkpointCount = count(unique quality checkpoint encoded tuples)`;
- `overallWorkflowAccuracy.total == checkpointCount`;
- separate safety namespace excluded from `checkpointCount`;
- diagnostic metrics excluded from all quality denominators.

This resolves the prior collision and primary-assignment ambiguity. The
remaining implementation obligation is mechanical: Phase 4 must enforce the
canonical tuple rules before scoring.

the initial fixture may remain approximately 15-20 focused cases, but it may not

Required coverage is present for:

- all-keep plan;
- review-only plan;
- all-deferred plan;
- all-rejected plan;
- integrity-blocked plan;
- mixed-blocked plan;
- mixed-blocked-review plan;
- ready-for-execution positive case;
- missing source action;
- unresolved policy action;
- structural overlap conflict;
- recommendation-scoped and memory-scoped warnings;
- unscoped warning integrity failure;
- duplicate recommendation normalization;
- conflicting duplicate recommendation integrity case;
- multi-call decision fingerprinting.

The blocker-code table correctly handles the ADR-004 catalog tension: six codes
are production-positive in the current planner, while `UNSUPPORTED_ACTION` and
`STALE_EVIDENCE` are reserved/not production-emittable for planner version `"2"`
and must be validated through required absence rather than invalid object
construction.

## Fixture Builder Boundary Verification

ADR-005 now restricts fixture builders to input construction only. It explicitly
forbids builders from:

- calling `plan_workflows()` to generate expected values;
- calling `apply_workflow_decisions()` to generate expected values;
- deriving expectations from actual planner output;
- computing expected aggregate statuses using production helpers;
- computing expected routing using production planner logic;
- importing committed benchmark artifacts as labels;
- importing evaluator comparison helpers to synthesize expected structures;
- bypassing production contracts with invalid enum values or model-construction
  shortcuts to manufacture planner outputs.

This is sufficient for ADR acceptance. Phase 4 should include direct import and
review checks to preserve this boundary in implementation.

## Safety Instrumentation Verification

Safety gates are now CI-checkable and bounded. Every safety property requires a
non-zero executed observation with `checksTotal >= 1`. The key audited areas are
covered:

| Safety area | Verification |
| --- | --- |
| `no_effects` | Bounded to the execution of `plan_workflows()` and `apply_workflow_decisions()`; checks filesystem writes, network/socket calls, subprocess/process launch, action execution APIs, persistence/scheduling APIs, and source object mutation |
| Source report immutability | Requires before/after snapshots of `AnalysisReport`, memories, recommendations, simulation report, policy/action data, plus sentinel proof that mutation would be detected |
| Deterministic planning | Requires at least two fresh equivalent inputs and canonical output comparison |
| Idempotent planning | Requires repeated same-process planning on fresh equivalent source objects |
| Atomic rejected decisions | Requires mixed invalid batch and byte-identical unchanged plan |
| Plan ID stability | Requires initial, valid-decision, and atomic-error stage comparison |
| Step/operation preservation | Requires canonical pre/post comparison after a valid state-changing decision |

The `no_effects` claim no longer overreaches. ADR-005 explicitly states that it
is a bounded CI assertion, not proof against every conceivable operating-system
effect. It also correctly scopes runner artifact writes outside the
planner/decision call boundary and under `examples/benchmarks/`.

## H1-H7 Verification

ADR-005 now includes a dedicated H1-H7 boundary table. It requires one positive
`ready_for_execution` case where all gates are true and at least one isolated
negative-control case per gate.

| Gate | Field or condition checked | Fixture pattern | Failure behavior | Verification |
| --- | --- | --- | --- | --- |
| H1 | At least one structural item with executable structural operation | Remove/suppress all structural items from a ready baseline | Records safety and aggregate-status failure identifying missing structural item | Pass |
| H2 | Every structural item eligible and approved | Leave one structural item unapproved or ineligible | Identifies structural item and unmet eligibility/approval field | Pass |
| H3 | Every non-structural actionable item resolved | Leave one non-structural actionable item unresolved | Identifies unresolved non-structural item | Pass |
| H4 | No unresolved review remains | Preserve one unresolved review item | Identifies review subtype and queue | Pass |
| H5 | No blocked item remains | Add item-level blocker where production construction allows isolation | Identifies blocked item and blocker code | Pass |
| H6 | No plan-level blocker remains | Add plan-level blocker while items otherwise eligible/resolved | Identifies plan-level blocker | Pass |
| H7 | Planning mode is `full` | Use `recommendations_only` with otherwise approved structural-looking work | Identifies planning mode | Pass |

Readiness cannot pass by vacuity under the hardened ADR. H5/H6 production
isolation difficulty is acknowledged without permitting invalid object
construction.

## Result-State Verification

ADR-005 now separates:

- invalid fixture: `evaluationStatus == "fixture_invalid"`;
- evaluator error: `evaluationStatus == "evaluator_error"`;
- scored checkpoint failure: `evaluationStatus == "scored_failure"`;
- safety failure: `evaluationStatus == "safety_failure"`;
- artifact generation failure: `evaluationStatus == "artifact_generation_failure"`;
- successful scored run: `evaluationStatus == "scored"`.

For invalid fixtures and evaluator errors, all metric names must be present with
`accuracy: null`, `passed: 0`, and `total: 0`. `gatePassed` short-circuits to
`false` unless `evaluationStatus == "scored"` and all quality, safety,
failure-list, and artifact clauses pass.

This resolves the prior result-contract ambiguity.

## Artifact Semantics Verification

ADR-005 now requires deterministic artifact semantics:

- JSON and Markdown render from one normalized result object;
- Markdown is checked against the source object rather than parsed as a second
  ground truth;
- volatile timestamps are excluded;
- absolute machine paths are excluded;
- platform-specific separators must not create drift;
- network dependency and external-data download are prohibited;
- external service data cannot enter workflow correctness;
- artifacts include metric numerators/denominators, safety results, diagnostic
  distributions, failures, fixture/planner versions, reproduction command,
  ground-truth statement, and planning-only disclaimer.

The semantic-equivalence rule is implementable and avoids brittle Markdown
round-tripping.
