## Executive Summary

V0.7 Phase 3 delivers a functional, evaluation-only simulation benchmark layer that loads `tests/fixtures/simulation_gold.json`, builds `AnalysisReport` inputs, invokes `simulate_recommendations()`, compares checkpoints, aggregates quality metrics, validates safety invariants, and publishes reproducible artifacts. The layer does not modify simulation semantics, pipeline behavior, or upstream governance/lifecycle/evolution/policy outputs.

Independent verification confirms:

- **Ground truth authority** is correctly scoped to `simulation_gold.json` only.
- **All eight quality metrics** are implemented and currently score `1.0000` on the 9-case fixture (119/119 checkpoints).
- **Safety gate** passes on all seven documented properties.
- **Diagnostic metrics** are separated and do not gate pass/fail.
- **Benchmark artifacts** (`simulation_evaluation.json`, `simulation_evaluation.md`) match live runner output.
- **17/17** evaluation tests pass (`python -m pytest tests/test_simulation_evaluation.py -q`).

Remaining gaps are **architectural/benchmark discipline weaknesses**, not scoring failures: orphan-metric definition drift from ADR-003, partial explainability checkpoint coverage, unused float-tolerance helpers, duplicated fixture builders (runner/test drift risk), and safety validation reimplemented inline rather than delegated to `tests/test_simulation.py` as the plan specified.

No high-severity release blockers were found.

**Verdict: APPROVED WITH LIMITATIONS**

**Estimated compliance: 91%**

### Medium Severity

#### M1 â€” Orphan merge accuracy metric definition diverges from ADR-003

**ADR-003 Â§6** defines `orphan_merge_accuracy` as **case-level**: `passed_orphan_cases / total_orphan_cases` (hard gate 1.0 on orphan scenarios).

**Implementation** computes checkpoint-level pass rate in the `orphan` category (currently 4/4 checkpoints on `sim_orphan_merge_no_keeper`), exposed via `CheckpointStats` (`simulation_evaluation.py:160â€“161`, `:653â€“672`).

**Impact:** Gate currently passes and orphan behavior is validated, but the reported metric is not strictly ADR-defined. Future partial orphan checkpoint failures could produce misleading orphan accuracy while case-level pass differs.

**Evidence:** ADR-003 lines 295â€“305; implementation `orphan: CheckpointStats(acc.orphan.passed, acc.orphan.total)` with 4 orphan-tagged checkpoints.

---

#### M2 â€” Explainability accuracy partially implemented vs ADR-003

**ADR-003** requires per simulated event:

- `explainability.recommendationId` non-empty
- `explainability.evidenceRefs` non-empty
- `explainabilitySource` gold matching when specified

**Implementation** checks only non-empty `evidenceRefs` and `resolution:` prefix (`simulation_evaluation.py:570â€“601`). Does not assert `recommendationId` non-empty or `explainabilitySource` against gold.

**Impact:** Explainability accuracy may report 1.0 while omitting ADR-specified dimensions. Current fixture passes implemented checks.

**Evidence:** `_evaluate_derived_checkpoints` explainability loop; no gold field for `explainabilitySource` in `simulation_gold.json`.

---

#### M3 â€” Metric consistency missing cluster removable-slot validation

**ADR-003 / Phase 3 plan Â§6** list `estimatedDuplicateReduction` alignment with cluster removable-slot math as a consistency check.

**Implementation** validates `estimatedDuplicateReduction` only when present as an exact integer in gold `expected`; no derived cluster-formula check across all cases.

**Impact:** Internal metric inconsistency on duplicate reduction could evade `metricConsistencyAccuracy` unless gold explicitly asserts the field.

**Evidence:** `metric_consistency` checks at lines 603â€“651; no cluster removable-slot computation.

---

#### M4 â€” Fixture builder duplication (plan non-compliance)

**Phase 3 plan Â§Evaluation Runner Plan** requires extracting `build_report_from_case()` into a shared module (e.g. `simulation_fixtures.py`) to prevent runner/test drift.

**Implementation** duplicates logic in `memd/benchmarks/simulation_evaluation.py:215â€“285` while `tests/test_simulation.py:39â€“101` retains a separate copy.

**Impact:** Future fixture or builder changes may desynchronize evaluation runner and property tests.

**Evidence:** Two independent `build_report_from_case` implementations; no `simulation_fixtures.py` module.

---

#### M5 â€” Safety validation not delegated to `test_simulation.py`

**Phase 3 plan Â§8** requires `evaluate_simulations()` to â€śinvoke or assert delegation to existing property tests in `test_simulation.py`.â€ť

**Implementation** reimplements safety inline in `_validate_safety()` without calling or importing property tests.

**Impact:** Dual maintenance paths; safety semantics could drift between evaluation runner and property test suite over time. Functionally equivalent on current fixture (both pass).

**Evidence:** Plan lines 409â€“413; `_validate_safety()` at lines 677â€“767 vs `tests/test_simulation.py` property tests.

---

#### M6 â€” Partial safety failure reporting vs ADR/plan

**ADR-003 / Phase 3 plan Â§8** state safety failures should appear in `failures[]` with `"category": "safety"`.

**Implementation** appends failures for some safety checks (source immutability, idempotency, monotonic reduction, no orphan removal) but **only** sets `safety.properties[...] = False` for orphan safety (partial), review preservation, and duplicate-removal safety without always appending failure records.

**Impact:** Artifacts may show `failures: []` while `safety.passed: false` in hypothetical failure scenarios â€” gate still blocks, but failure detail incomplete.

**Evidence:** `_validate_safety()` orphan/review/duplicate branches vs idempotency branch that appends failures.

---

#### M7 â€” Float tolerance constants declared but largely unused

**Plan Â§7** defines Â±0.01 for percentages and Â±0.0001 for utilization rates.

**Implementation** defines `PERCENT_TOLERANCE = 0.01`, `RATE_TOLERANCE = 0.0001`, and `_floats_close()` (`:33â€“34`, `:292â€“293`). Only `PERCENT_TOLERANCE` is used (trusted gain bound at `:612â€“618`). `RATE_TOLERANCE` and `_floats_close` are never called. Gold integer comparisons use exact `==`.

**Impact:** Documented tolerance policy is partially implemented. Not a current scoring defect because gold uses exact integers and utilization is diagnostic-only.

**Evidence:** Grep shows `_floats_close` defined once, never invoked; tests only assert constant values exist.

### Low Severity

#### L1 â€” Per-dimension checkpoint totals exceed overall total

Per-dimension passed/total in artifacts sum to 124 (18+18+25+14+4+9+36) while overall is 119, because checkpoints are multi-tagged. Accurate but may confuse operators interpreting dimension tables.

**Evidence:** `simulation_evaluation.md` Per-Dimension Accuracy table vs Summary 119/119.

---

#### L2 â€” Review preservation safety scoped to single case

`review_preservation` safety property validated only on `sim_review_1` (`_validate_safety:757â€“763`), not parametrized across all gold cases with review resolutions. Broader coverage exists in `tests/test_simulation.py` but outside evaluation runner.

---

#### L3 â€” Diagnostic gold fields not recorded in diagnostic output

Gold cases include diagnostic fields `implicitKeepFallback` and `resolutionFallbackExplainability` (e.g. `sim_keep_1`, `sim_implicit_keep_fallback`). Evaluation correctly excludes them from gating but does not record whether the fallback path was exercised in diagnostic distributions.

---

#### L4 â€” Package version vs benchmark version mismatch in artifacts

Committed JSON shows `version: "0.7.0"` and `metadata.memdVersion: "0.6.0"`. Not incorrect, but may confuse release tracking.

---

#### L5 â€” Missing negative-path tests for safety gate failure

`test_evaluate_simulations_safety_gate` only asserts happy-path pass. No test verifies that a safety property failure sets `gate_passed = False`.

---

#### L6 â€” Regression guards documented but not executed

LongMemEval/PERMA guards are correctly marked non-gating and `passed: null`. Plan allows deferral at launch; no automated posture guard smoke in CI.

## Recommendation

**V0.7 Release Readiness Audit may begin.**

Phase 3 evaluation meets the quantitative and architectural requirements to unblock V0.7 simulation-track sign-off and V0.8 workflow planning. Additional hardening is **recommended but not required** before release:

1. Align `orphanMergeAccuracy` with ADR-003 case-level definition (1/1).
2. Extract shared `build_report_from_case()` to prevent runner/test drift (M4).
3. Complete explainability and metric-consistency checks per ADR-003 (M2, M3).
4. Ensure all safety failures populate `failures[]` with `category: "safety"` (M6).
5. Wire or document relationship between evaluation runner safety and `tests/test_simulation.py` property tests (M5).
6. Use or remove unused `RATE_TOLERANCE` / `_floats_close` (M7).

None of these items block benchmark certification or V0.7 release readiness on the current labeled fixture.
