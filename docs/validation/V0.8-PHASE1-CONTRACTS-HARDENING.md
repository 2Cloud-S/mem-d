## 2. M1 Interpretation and ADR Justification

Audit M1 originally suggested rejecting review queues on non-review actions. That interpretation was **not implemented** because it would violate ADR-004.

In ADR-004, recommendation action and review requirement are separate dimensions. Merge and archive recommendations may legitimately require review due to:

- `requiresHumanApproval`
- policy `requires_review`
- planner status `requires_review`
- review action semantics

Implemented M1 rule (consistency only, no derivation):

1. `action == review` → `reviewRequirement.required` must be `true`
2. `requiresHumanApproval == true` → `reviewRequirement.required` must be `true`
3. `plannerItemStatus == requires_review` → `reviewRequirement.required` must be `true`
4. `policyDecision == requires_review` → `reviewRequirement.required` must be `true`
5. Merge/archive items may still carry required review requirements
6. Inconsistent combinations raise `ValidationError` (no coercion)

## 4. Tests Added

### M1 tests

- review action with `required=False` → invalid
- `requiresHumanApproval=True` with `required=False` → invalid
- planner status `requires_review` with `required=False` → invalid
- policy decision `requires_review` with `required=False` → invalid
- merge with valid required review → valid
- archive with valid required review → valid
- proposed merge without review signals → valid
- blocked item without review requirement → valid

### M2 tests

Parametrized invalid enum rejection via `model_validate` for:

- planner item status (item + step)
- operator item status
- aggregate plan status
- planner status
- planning mode
- workflow step type (step + operation)
- blocker code
- review subtype
- review queue ID
- approval decision type
- recommendation action
- action priority
- policy decision

### L1 tests

- whitespace-only keeper/removable/archive/review targets rejected
- duplicates after trimming rejected
- normalized valid IDs accepted
