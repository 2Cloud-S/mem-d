# V0.8 ADR-005 Workflow Evaluation Architecture Audit

Date: 2026-06-23  
Type: Independent architecture audit  
Scope: ADR-005 and repository evidence; audit only

## 1. Executive Summary

ADR-005 establishes the correct evaluation boundary for V0.8 Phase 4. It keeps
workflow evaluation separate from recommendation and simulation evaluation,
names `tests/fixtures/workflow_gold.json` as the exclusive workflow correctness
authority, evaluates initial planning separately from decision application, and
requires exact quality scores plus independent safety gates.

The architecture is substantially aligned with production planner version `"2"`.
The audit confirmed alignment with the implemented APIs, aggregate-status
precedence, H1-H7 readiness logic, policy aggregation, structural-overlap
routing, warning scoping, integrity blocking, cumulative decision-state
fingerprinting, and pipeline/reporting boundaries.

No high-severity defect, ground-truth contamination, execution authorization, or
upstream rescoring was found. Four medium findings prevent unambiguous Phase 4
implementation:

1. the 18-case inventory does not normatively map every aggregate status and
   every blocker enum member to an attainable scored checkpoint;
2. safety-property coverage can pass without a specified minimum number of
   observations or controls, and `no_effects` lacks explicit instrumentation
   boundaries;
3. checkpoint semantic keys have no canonical encoding or escaping rule and the
   primary-metric assignment is not represented in the conceptual fixture;
4. an invalid fixture is said to return all metrics without defining the metric
   value and denominator representation for that unscored state.

Three low findings concern explicit negative-control coverage, provenance
completeness metadata, and artifact semantic-comparison detail.

**Final verdict: NOT APPROVED**  
**Estimated compliance: 88%**

ADR-005 should not become Accepted until the medium findings are dispositioned.
Phase 4 correction and implementation planning may begin. Phase 4 implementation
should not begin against the current text because two implementers could produce
different valid-looking fixtures, checkpoint IDs, and invalid-result payloads.

## 3. Architecture Alignment

### Production API and version alignment

| Concern | Production evidence | ADR-005 assessment |
| --- | --- | --- |
| Planner version | `memd/workflows.py:66` sets `WORKFLOW_PLANNER_VERSION = "2"` | Correctly states `"2"` |
| Initial planning | `plan_workflows(report, *, options: PlanningOptions | None = None)` at `memd/workflows.py:1864` | Correctly evaluates initial planning; uses ellipsis rather than redefining the signature |
| Decisions | `apply_workflow_decisions(plan, decisions)` at `memd/workflows.py:1906` | Correctly evaluates a separate pure transformation |
| Decision error | `WorkflowDecisionError` and pre-update batch validation | Correctly requires atomic failure and unchanged plan |
| Pipeline boundary | Simulation attached first, then `plan_workflows(report)` at `memd/pipeline.py:88-90` | Correctly describes post-simulation initial-plan attachment |
| Report boundary | `workflowPlan` serialization/rendering in `memd/reports.py`; file writing is a separate reporting helper | Correctly excludes reporting writes from planner behavior |

ADR-005 does not repeat ADR-004's obsolete conceptual `include_keep` keyword
signature. Its fixture uses the production `PlanningOptions.includeKeep` field,
and its API intentionally leaves construction details to Phase 4.

### Aggregate status and readiness

`compute_aggregate_status()` implements the ADR-004 first-match order:

```text
integrity_blocked
empty
all_keep
all_blocked
mixed_blocked_review
mixed_blocked
rejected
deferred
ready_for_execution
partially_approved
approved
requires_review
proposed
```

The implemented `handoff_ready` expression at `memd/workflows.py:1804-1814`
matches H1-H7: non-empty structural work, all structural items eligible, all
non-structural items resolved, no unresolved review, no blocked item, no
plan-level blocker, and full planning mode. ADR-005 correctly requires a positive
readiness case and one perturbation per H1-H7 condition.

### Policy and integrity behavior

`_aggregate_policy()` at `memd/workflows.py:547-575` implements:

```text
blocked > requires_review > unresolved existing action > approved
```

Missing source action IDs are not treated as unresolved policy; `_pass_b()` adds
integrity causes, blocks the affected item, withholds structural steps, and emits
a plan-level `INPUT_INTEGRITY` blocker. ADR-005 correctly distinguishes those
paths.

### Review routing, warning scope, and overlap

- `PRIMARY_QUEUE_BY_SUBTYPE` implements simulation safety, conflict, policy,
  unknown category, lifecycle, low trust, and general precedence.
- All eleven `ReviewSubtype` enum members are represented in production routing.
- Recommendation-scoped and memory-scoped `DUPLICATE_REMOVAL_SKIPPED` warnings
  attach only to matching items; an unscoped warning becomes an integrity cause.
- `_apply_structural_overlap_conflicts()` runs before frozen item construction,
  routes implicated archive/merge items to conflict review, and suppresses their
  structural steps.

ADR-005 describes these verified semantics without changing them.

### Decision fingerprint

`_decisions_fingerprint()` hashes every item's resulting
`workflowItemId`/`operatorItemStatus` pair, sorted by item ID. Repeated calls
therefore represent cumulative resulting operator state, not merely the latest
batch. `apply_workflow_decisions()` leaves `workflowPlanId`, steps, operations,
blockers, and evidence unchanged. ADR-005 is correctly aligned.

### Identified terminology/API mismatches

No incorrect production field, enum value, callable name, or planner version was
found in ADR-005. The blocker-catalog coverage gap in Finding M1 is not a naming
mismatch; it is a mismatch between the desired evaluation coverage and currently
reachable production behavior.

## 4. Ground-Truth and Fixture Validation

ADR-005 is explicit that `tests/fixtures/workflow_gold.json` is the sole workflow
correctness authority. It separately identifies:

- recommendation gold as recommendation-only;
- simulation gold as simulation-only;
- external validation as non-gating evidence;
- unit/integration tests as implementation assurance;
- generated benchmark artifacts as reports, not labels.

No mechanism permits upstream accuracy, external distributions, or existing
artifacts to enter a workflow numerator, denominator, threshold, or safety gate.

Fixture builders are constrained appropriately in principle: they construct
inputs only, must be pure/local/deterministic, must expose reproducible expanded
inputs, and must not derive expectations from the planner. That prevents builders
from becoming a hidden second correctness authority.

The remaining builder risk is enforcement. Phase 4 should include a test or
code-review rule that builder modules do not import `plan_workflows()`,
`apply_workflow_decisions()`, evaluator comparison helpers, or committed result
artifacts when constructing expected values. This is a low-level implementation
guard, not a ground-truth defect in ADR-005.

## 7. Checkpoint Accounting Validation

The proposed identity shape is sound at a conceptual level:

```text
{caseId}::{phase}::{checkpointType}::{semanticKey}
```

ADR-005 correctly requires:

- global uniqueness after expansion;
- duplicate-ID rejection before evaluation;
- exactly one primary dimension metric;
- set-union accounting for overall accuracy;
- `checkpointCount == overallWorkflowAccuracy.total`;
- a separate safety namespace excluded from quality denominators;
- no hidden run-dependent derived checks.

The missing detail is canonical component encoding. A case ID, phase, or semantic
key containing `::`, or two different semantic objects rendered to the same
informal key, can collide. The document also says a checkpoint “has” a primary
dimension but does not show whether that assignment is explicit fixture data or
a total mapping from `checkpointType`. Finding M3 requires a canonical tuple
encoding or length-prefixed/escaped serialization and a closed mapping from every
checkpoint type to exactly one metric.

All 18 properties are said to be independent hard gates. Diagnostics cannot
override them. However, ADR-005 does not define a non-zero observation count per
safety property or require every property key to be present in `safetyResults`.
An implementation could report a property as passed without executing a check.
That is Finding M2.

`no_effects` cannot prove the absence of all effects in an absolute sense. A
realistic CI assertion is bounded: while calling `plan_workflows()` and
`apply_workflow_decisions()`, deny or spy on filesystem mutation APIs, socket and
HTTP creation, subprocess/process-launch APIs, and known persistence adapters;
snapshot the designated workspace/temp roots; and assert source objects are
unchanged. Artifact writes must occur after leaving this boundary. ADR-005 hints
at instrumentation but does not specify these boundaries or acknowledge that the
claim is bounded.

### H1-H7

The required boundary-perturbation matrix is the correct approach. A valid full
plan establishes readiness, then exactly one predicate is changed per negative
variant:

| Gate | Isolated perturbation |
| --- | --- |
| H1 | Remove all structural items while resolving non-structural work |
| H2 | Leave one structural item unapproved or ineligible |
| H3 | Leave one non-structural actionable item unresolved |
| H4 | Add one unresolved review item |
| H5 | Add one blocked item without adding a plan blocker |
| H6 | Add one plan-level blocker while items otherwise remain eligible |
| H7 | Change only planning mode to `recommendations_only` |

For H5 and H6, fixture construction must preserve isolation; ordinary production
inputs may cause both predicates simultaneously. The expected expanded fixture
should document which exact predicate changed. With that requirement, H1-H7 are
non-vacuous and independently testable.

## 9. Identity and Provenance Validation

ADR-005 correctly separates exact structural output from relational identity
assertions. It requires lowercase 64-character SHA-256 form, repeat equality,
canonical-order equality, safety-relevant inequality, stable `workflowPlanId`
across decisions, and cumulative `decisionsFingerprint` changes. Complete literal
hashes require an explicit stability reason.

This matches production behavior:

- plan identity hashes planner version, options, source fingerprint, simulation,
  policy, recommendations, resolutions, and safety-relevant signals;
- item, blocker, and step IDs derive deterministically from plan identity and
  semantic inputs;
- decision fingerprint hashes cumulative item operator states;
- decisions do not alter plan identity.

Provenance comparison is also appropriately layered: exact expected item refs,
resolvable plan closure, item evidence as a subset, warning scope, and dedicated
cases with complete closure. Digest references for conflicting duplicate
variants match the verified implementation and do not require raw duplicate
payload copies.

The fixture should explicitly mark a provenance expectation as `partial` or
`complete`; relying only on the presence of selected optional fields makes it
hard for the evaluator to know whether unexpected plan-level provenance is a
failure. This is Finding L2.

## 10. Evaluation API and Result Contract

The proposed `evaluate_workflows(gold_path: Path | None = None)` is consistent
with established evaluators and correctly returns a result rather than writing
artifacts. The runner owns JSON/Markdown output.

Positive attributes:

- all eleven metrics are required in `qualityMetrics`;
- each contains accuracy, passed, and total;
- safety is separate;
- failures have explicit kinds and structured checkpoint details;
- fixture-invalid and scored-failure runs both fail the gate;
- result and evaluator are intended to be immutable/deterministic;
- planner version and ground-truth authority are explicit.

The gate formula is semantically clear:

```text
gatePassed =
    all 11 quality accuracies == 1.0
    and all safety properties passed
    and failures == []
```

The invalid-fixture representation is not implementable without a choice that
the ADR has not made. ADR-005 requires every metric to be represented, forbids a
zero denominator from being scored or omitted, and says an invalid fixture may
be unscored. It does not state whether `accuracy` becomes `null`, whether
`passed/total` are `0/0`, or whether a distinct `evaluationStatus` is required.
Finding M4 requires one exact representation and an exact formula that short
circuits on fixture validity.

## 11. Artifact and CI Validation

ADR-005 requires deterministic JSON and Markdown artifacts generated from the
same evaluation API used by tests. It correctly prohibits volatile timestamps,
absolute paths, platform-specific separators, network access, external downloads,
threshold tuning, planner monkeypatching, and auto-rebaselining on planner-version
change.

The proposed artifacts include every required quality, safety, diagnostic,
failure, version, reproduction, authority, and planning-only field. Test/runner
drift is addressed by requiring both to call the public evaluation API and by
checking artifact regeneration.

“Semantic equivalence” between JSON and Markdown needs an implementation rule.
Recommended default: build one normalized result dictionary, serialize JSON from
it, render Markdown from it, and test the Markdown renderer against that same
dictionary rather than parse Markdown back into a second result model. This is
Finding L3.

## 12. Scope Boundary Validation

ADR-005 introduces none of the prohibited capabilities:

- no workflow or action execution;
- no memory mutation;
- no plan or decision persistence;
- no scheduling, retry, or rollback;
- no network dependency;
- no approval UI;
- no threshold tuning;
- no planner behavior mutation;
- no recommendation or simulation rescoring;
- no external-data correctness score.

The only file-writing allowance is runner-owned benchmark artifact generation
after evaluation. This is consistent with existing evaluation patterns and is
not workflow execution.

## 13. Findings

### High

None.

### Medium

#### M1 — Aggregate-status and blocker-catalog coverage is not normative

**Evidence:** ADR-005 provides 18 descriptive cases but does not map each
`PlanAggregateStatus` enum member to a required checkpoint. It also does not map
each `WorkflowBlockerCode` to positive/negative coverage. Production emits six
codes. `UNSUPPORTED_ACTION` is reserved because all values of the closed
`RecommendationAction` enum have explicit branches (`memd/workflows.py:1-7`),
and `STALE_EVIDENCE` has no emission path. ADR-004 nevertheless lists both codes.

**Architectural impact:** Phase 4 cannot both “evaluate production behavior” and
positively produce every blocker code without inventing invalid inputs or using
contract bypasses. Aggregate-status completeness also depends on inference from
case descriptions, risking a zero or accidental denominator for a status path.

**Required disposition:** Add a normative coverage table for all 13 aggregate
statuses and all eight blocker codes. For each blocker, mark it as production
positive, required absence, reserved/not-applicable, or architecture-blocking.
Do not use `model_construct()` or invalid-enum bypasses to manufacture planner
outputs. Explicitly disposition `UNSUPPORTED_ACTION` and `STALE_EVIDENCE` against
the current version-2 planner.

**Blocks ADR acceptance:** Yes  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** Yes

#### M2 — Safety coverage cardinality and effect boundaries are underspecified

**Evidence:** All 18 safety properties are hard gates, but the ADR does not
require a non-zero check count, an executed observation, or required control per
property. `no_effects` says to instrument or isolate boundaries “sufficiently”
without defining the bounded APIs/roots or acknowledging proof limits.

**Architectural impact:** A result could report every property as passed without
testing it, or make incompatible claims about global absence of effects. Several
properties could pass vacuously on cases that never exercise their precondition.

**Required disposition:** Define a safety result record with `passed`,
`checksPassed`, `checksTotal`, and failure details; require `checksTotal >= 1` for
every property. Add explicit controls where listed in Section 8. Bound
`no_effects` to planner/decision calls with filesystem mutation, network/socket,
subprocess/process-launch, and persistence sentinels plus workspace/temp and
source snapshots. Require a sentinel self-test so the harness is known to detect
an attempted effect.

**Blocks ADR acceptance:** Yes  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** Yes

#### M3 — Checkpoint key encoding and primary-metric assignment are ambiguous

**Evidence:** The checkpoint shape concatenates four free-form components with
`::`. No character restrictions, escaping, length-prefixing, canonical JSON
tuple encoding, or collision procedure is defined. The conceptual fixture does
not show a checkpoint type or primary-metric assignment for expected leaves,
although the text requires exactly one primary metric.

**Architectural impact:** Distinct expectations can collide, and independent
implementations can assign the same expected field to different metric
denominators. Duplicate detection does not prevent collisions if two semantic
objects first collapse to the same string.

**Required disposition:** Define checkpoint identity as a canonical encoded
tuple (recommended: canonical JSON array of the four normalized components,
optionally hashed only for storage) or define strict escaping. Publish a closed,
total `checkpointType -> primaryMetric` mapping. Expanded fixture validation must
reject unknown types, unmapped types, duplicate encoded tuples, and multiple
primary assignments.

**Blocks ADR acceptance:** Yes  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** Yes

#### M4 — Invalid-fixture result representation is incomplete

**Evidence:** ADR-005 requires all eleven metrics to appear, says zero
denominators are invalid and never scored or omitted, and distinguishes invalid
fixtures from scored failures. It does not define the metric values for an
unscored invalid fixture.

**Architectural impact:** JSON schema, Markdown rendering, and `gatePassed`
calculation can diverge. One implementation may emit `0.0`, another `null`, and
another throw before returning a result.

**Required disposition:** Add `evaluationStatus` (recommended values `scored` and
`fixture_invalid`). For invalid fixtures, require all metric entries with
`accuracy: null`, `passed: 0`, `total: 0`, explicit fixture failures, and
`gatePassed: false`; or choose another single exact representation. Define
`gatePassed` to require `evaluationStatus == scored` before evaluating metric and
safety clauses.

**Blocks ADR acceptance:** Yes  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** Yes

### Low

#### L1 — Some case controls are implied rather than required

**Evidence:** Warning-scoping cases explicitly imply unrelated-item behavior,
but structural overlap does not normatively require both a non-overlap control
and an unrelated unaffected item. Immutability/effect sentinels also lack harness
self-tests.

**Architectural impact:** A globally over-blocking planner could satisfy the
positive conflict expectation.

**Required disposition:** Make the controls in Section 8 mandatory fixture or
harness checks.

**Blocks ADR acceptance:** No  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** No, if addressed during implementation plan

#### L2 — Provenance partial versus complete mode should be explicit data

**Evidence:** The ADR describes partial expected blocks and dedicated complete
closure cases but does not show a `matchMode` or equivalent field.

**Architectural impact:** Unexpected provenance may be inconsistently accepted
or rejected.

**Required disposition:** Add an explicit `partial`/`complete` provenance match
mode, defaulting dedicated provenance cases to complete.

**Blocks ADR acceptance:** No  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** No

#### L3 — JSON/Markdown semantic equivalence mechanism is unspecified

**Evidence:** CI must verify semantic equivalence, but Markdown is not defined as
a parseable result contract.

**Architectural impact:** Teams may introduce a brittle Markdown parser or allow
renderer drift.

**Required disposition:** Render both artifacts from one normalized result and
test required Markdown sections/values against that source object.

**Blocks ADR acceptance:** No  
**Blocks Phase 4 planning:** No  
**Blocks Phase 4 implementation:** No

## 14. Risks and Limitations

- The proposed 18 cases are architecture-compliance examples, not empirical
  production accuracy evidence.
- Composite matrix cases can obscure which exact precondition failed unless the
  expanded checkpoint plan is emitted in failures.
- Planner version changes can invalidate fixture expectations; ADR-005 correctly
  forbids automatic rebaselining.
- External validation remains useful posture evidence but has no labeled
  workflow outcomes and cannot close fixture gaps.
- Relational identity tests reduce brittle hashes but still require carefully
  isolated safety-relevant input mutations.
- H5 and H6 are difficult to perturb independently through ordinary production
  inputs; fixture construction must document isolation without redefining the
  aggregate function.
- An effect sentinel proves only the instrumented boundary, not the absence of
  every conceivable operating-system effect.

## 15. Acceptance and Phase Gate Decisions

### ADR-005 may become Accepted

**No.** Findings M1-M4 require architecture-level dispositions first.

### Phase 4 planning may begin
